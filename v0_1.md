# Solvent v0.1 Technical Implementation Plan

**Scope:** first runnable backend slice  
**Target command:** `solvent run --agent stub --seed 42`  
**Primary output:** a deterministic episode trace, ledger, and terminal run summary  
**Out of scope:** score decomposition beyond basic run accounting, omniscient/realizable reference policies, reactive market dynamics, multi-task markets, trace viewer, SWE-Lancer, live LLM agents

---

## 1. v0.1 Objective

v0.1 should prove the core environment loop works end to end:

1. Generate a seeded synthetic freelance market with one `data_clean` job type.
2. Expose only public job information through a tool-style API.
3. Let a stub harness inspect, bid, submit an artifact, and end ticks.
4. Resolve bid acceptance deterministically against a hidden `reservation_price`.
5. Verify submitted work with deterministic checks.
6. Account for starting balance, tool/model burn, per-tick overhead, accepted-job revenue, and insolvency/turn-cap termination.
7. Write an append-only JSONL trace that can later support scoring, replay, and the trace viewer.

The goal is not to make an intelligent agent yet. The goal is to establish the boundary between harness and environment and make every economic state transition auditable.

---

## 2. Key Design Decisions

### 2.1 Use `reservation_price` Consistently

The customer-side hidden price should be named `reservation_price` in code and docs. It means:

> The maximum bid this deterministic customer accepts for this generated job.

It is not claimed to estimate real-world customer demand. v0.1 is a designed market with known hidden ground truth.

### 2.2 Keep the Harness Boring

The first harness should be a deterministic stub, not an LLM harness. It exists to exercise the environment:

- list jobs
- inspect the first job
- bid a configured price
- submit either a valid or invalid cleaned CSV
- end ticks until termination

This makes failures environmental rather than model-induced.

### 2.3 Make the Trace the Source of Truth

Every meaningful state transition should emit one JSONL event. The in-memory `Episode` is useful, but the trace is the durable artifact.

Required event properties:

- deterministic ordering
- sufficient hidden/public separation
- enough data to recompute basic ledger state
- no hidden ground truth leaked in harness observations

### 2.4 Implement One Task Deeply

`data_clean` should be implemented well enough to validate the verifier pattern:

- seeded raw CSV input
- public brief with schema requirements
- hidden rubric/checks
- submitted artifact as CSV text
- deterministic pass/fail result with failed check names

No `copywriting`, `extract`, or `code_fix` in v0.1.

### 2.5 Static Board Now, Reactive Dynamics Later

The current parent TDD keeps reactive market dynamics in the design because they are load-bearing for later VB-like quality and Phase 3 RL. v0.1 should **not** build them. v0.1 is the first-demo static version:

- fixed seeded board
- deterministic arrivals
- no reputation state
- no expiry windows
- no path-conditioned market sampling beyond bid-attempted/accepted/submitted job state

This is intentional scope discipline. The trace and environment boundaries should still leave room for v0.4+ reactivity: `arrival_tick` exists, jobs are exposed through `Market.available_jobs(tick)`, and events are ordered by tick. The future reactive transition function can replace the market internals without changing the harness/tool boundary.

## 4. Module Responsibilities

### 4.1 `solvent.env.models`

Define the shared dataclasses and enums.

Core types:

```python
@dataclass(frozen=True)
class EnvConfig:
    seed: int
    config_id: str
    start_balance: Decimal
    horizon_ticks: int
    overhead_per_tick: Decimal
    tool_call_cost: Decimal
    trace_path: Path

@dataclass(frozen=True)
class JobPublic:
    id: str
    type: str
    brief: str
    inputs: dict[str, Any]
    arrival_tick: int

@dataclass(frozen=True)
class Job:
    id: str
    type: str
    brief: str
    inputs: dict[str, Any]
    arrival_tick: int
    reservation_price: Decimal
    est_cost: Decimal
    rubric: Rubric

@dataclass(frozen=True)
class CheckResult:
    name: str
    passed: bool
    message: str

@dataclass(frozen=True)
class VerifyResult:
    passed: bool
    score: float
    checks: list[CheckResult]

@dataclass(frozen=True)
class Event:
    tick: int
    kind: str
    payload: dict[str, Any]
    balance_after: Decimal
    burn_delta: Decimal

# ---- types referenced above; held in-env, never serialized to the trace ----

@dataclass(frozen=True)
class Check:
    name: str
    # (artifact, inputs) -> CheckResult; callable predicate, in-env only
    predicate: Callable[[str, dict[str, Any]], CheckResult]

@dataclass(frozen=True)
class Rubric:
    checks: tuple[Check, ...]   # ordered, named checks; lives on the hidden Job

# ---- mutable env bookkeeping + run result ----

@dataclass
class AcceptedJob:
    job_id: str
    contract_price: Decimal     # the accepted bid price (final)
    submitted: bool = False
    paid: bool = False

@dataclass(frozen=True)
class EpisodeSummary:
    seed: int
    config_id: str
    start_balance: Decimal
    end_balance: Decimal
    net_revenue: Decimal        # end_balance - start_balance
    terminated_reason: str      # "insolvent" | "turn_cap"
    ticks_elapsed: int
    jobs_paid: int
    trace_path: Path
```

Implementation notes:

- Use `Decimal` for money, serialized as strings in JSONL.
- Keep `Job` hidden inside the environment.
- Convert to `JobPublic` before returning anything to the harness.
- Do not put callable predicates directly into JSON-serialized structures.
- `Rubric`/`Check` carry callables and live only on the hidden `Job`; they are never serialized.
- `config_id` is the canonical run label used in `EpisodeSummary` and CLI output, e.g. `stub:happy_path`.
- Define `VerifyResult.passed = all(c.passed for c in checks)` and `VerifyResult.score = passed_checks / total_checks` (graded in `[0, 1]` now so it can feed RL reward shaping later).

### 4.2 `solvent.env.clock`

Own tick progression and horizon termination.

Responsibilities:

- initialize at tick `0`
- expose `current_tick`
- advance one tick
- report `is_turn_cap_reached`

Expected API:

```python
class Clock:
    def __init__(self, horizon_ticks: int): ...
    @property
    def tick(self) -> int: ...
    def advance(self) -> int: ...
    def reached_horizon(self) -> bool: ...
```

Tests:

- starts at `0`
- advances deterministically
- horizon reached exactly at configured cap

### 4.3 `solvent.env.ledger`

Own all monetary state.

Responsibilities:

- track `start_balance`
- track current `balance`
- debit burn
- debit overhead
- credit revenue
- prevent hidden mutation outside ledger methods
- report insolvency

Expected API:

```python
class Ledger:
    def __init__(self, start_balance: Decimal): ...
    @property
    def balance(self) -> Decimal: ...
    def debit_burn(self, amount: Decimal) -> Decimal: ...
    def debit_overhead(self, amount: Decimal) -> Decimal: ...
    def credit_revenue(self, amount: Decimal) -> Decimal: ...
    def insolvent(self) -> bool: ...
```

Event emission can either live inside `Environment` or be coordinated by `Environment` after ledger operations. Prefer the latter in v0.1 so `Ledger` remains simple and testable.

Tests:

- credits and debits are exact
- insolvency when balance is `<= 0`
- no float rounding errors

### 4.4 `solvent.trace.writer`

Append normalized JSON objects to disk.

Responsibilities:

- create parent directories
- write one JSON object per line
- flush after each event
- serialize `Decimal`, `Path`, dataclasses, and simple nested structures safely

Expected API:

```python
class TraceWriter:
    def __init__(self, path: Path): ...
    def write(self, event: Event) -> None: ...
    def close(self) -> None: ...
```

Serialization rules:

- `Decimal` as string, e.g. `"19.95"`
- event `kind` as stable snake-case string
- no Python reprs
- no hidden fields in events generated for public observations, unless the event kind is explicitly internal
- no wall-clock timestamps in v0.1 traces (keeps the seed-42 trace byte-identical across runs for snapshot testing)

Required v0.1 event kinds. **Names are canonical — they match the parent TDD §C.5 so the v0.2 scorer reads v0.1 traces unchanged.** Do not invent synonyms.

Economic / action events (from TDD C.5):

- `board_seen`
- `inspected`
- `bid_made`
- `bid_accepted`
- `bid_declined`
- `submitted`
- `verified_pass`
- `verified_fail`
- `paid`
- `overhead_charged`
- `terminated`

Lifecycle events (v0.1 additions, not in C.5; non-economic markers):

- `episode_started`
- `tick_advanced`
- `invalid_action`

Burn is **not** a standalone event: it is recorded as `burn_delta` on the tool-call event that incurred it (see §7). There is no `burn_charged` kind.

### 4.5 `solvent.tasks.data_clean`

Generate and verify data-cleaning jobs.

v0.1 job shape:

- input CSV contains rows with inconsistent whitespace, missing values, and mixed casing
- public brief asks for a cleaned CSV with a specific schema
- output must normalize column names, trim whitespace, normalize casing, and drop invalid rows according to the brief

Example public input:

```csv
full_name,email,signup_date,plan
" Ada Lovelace ",ADA@EXAMPLE.COM,2026-01-04, Pro
"Grace Hopper",grace@example.com,2026-01-05,enterprise
"Missing Email",,2026-01-06,free
```

Example required output schema:

```text
name,email,signup_date,plan
```

Verifier checks:

- CSV parses successfully
- exact required columns in exact order
- row count matches expected valid rows
- no blank required cells
- emails lowercased
- names trimmed
- `plan` is one of `free`, `pro`, `enterprise`
- no extra columns

Expected API:

```python
def generate_data_clean_job(seed: int, job_index: int) -> Job: ...
def verify_data_clean(job: Job, artifact: str) -> VerifyResult: ...
```

The generator should produce deterministic variation by seed but keep difficulty low. v0.1 should test the loop, not model robustness.

Tests:

- generated job is stable for same seed/index
- valid artifact passes all checks
- malformed CSV fails parse check
- wrong columns fail schema check
- blank required cell fails completeness check

### 4.6 `solvent.env.market`

Generate and expose jobs by tick.

v0.1 can generate a small fixed-size board:

- 3 `data_clean` jobs per episode
- arrival ticks: `0`, `1`, `2`
- deterministic `reservation_price` and `est_cost`
- no decoys required yet, but allow negative-value jobs in the model for v0.2

Expected API:

```python
class Market:
    def __init__(self, seed: int): ...
    def available_jobs(self, tick: int) -> list[Job]: ...
    def get_job(self, job_id: str) -> Job: ...
```

Board semantics: `available_jobs(tick)` returns every job with `arrival_tick <= tick` (a cumulative board, not just jobs that arrived on this exact tick). The `Market` is stateless about progress; removing bid-attempted, accepted, or submitted jobs from the *public* board is the `Environment`'s responsibility (§4.8), and in-progress work is surfaced only via `list_in_progress`.

Tests:

- same seed yields same jobs
- different seeds yield different job ids or inputs
- arrival filtering works
- unknown job id raises a typed error

### 4.7 `solvent.env.tools`

Define action/result shapes, even if implemented as direct method calls in v0.1.

Action types:

```python
ListJobs
InspectJob(job_id)
Bid(job_id, price)
Submit(job_id, artifact)
CheckBalance
ListInProgress
EndTick
```

`respond` and `clarify` (TDD C.5) are intentionally **deferred to v0.3** along with revisions and manipulation events; they are absent from the v0.1 action space.

Tool result types can be dataclasses or plain dictionaries, but the environment should normalize them.

This module prevents the harness from calling arbitrary environment internals.

### 4.8 `solvent.env.env`

Coordinate the episode.

Responsibilities:

- initialize `Clock`, `Ledger`, `Market`, and `TraceWriter`
- expose public observation/tool methods
- charge tool-call burn
- emit trace events
- maintain accepted jobs and bid prices
- route submissions to the correct verifier
- credit revenue only after a passing verifier
- advance ticks and overhead
- terminate on insolvency or turn cap

State to track:

```python
bid_jobs: set[str]
accepted_jobs: dict[str, AcceptedJob]
submitted_jobs: set[str]
terminated_reason: str | None
```

Important invariants:

- A job cannot be submitted before an accepted bid.
- A job cannot be paid twice.
- A declined bid does not create in-progress work.
- **Each job accepts at most one bid; bids are final.** Once a bid is made on a job (accepted or declined) the job cannot be re-bid. This closes the exploit where an agent binary-searches the hidden `reservation_price` to capture full surplus every time — which would nullify pricing regret, the flagship signal.
- The public board (`list_jobs`) excludes every job in `bid_jobs`, including declined jobs, accepted jobs, and submitted jobs. In-progress accepted work is visible only via `list_in_progress`.
- A harness never receives `reservation_price`, `est_cost`, rubric internals, or expected output.
- Every tool call charges burn exactly once.
- `end_tick` charges overhead and advances the clock.
- Termination is evaluated immediately after any ledger debit/credit (insolvency: `balance <= 0`) and after `end_tick` (turn cap: `tick >= horizon_ticks`).
- `finalize()` is idempotent and emits `terminated` exactly once. Only the environment owner calls it — the harness never does.

Suggested method surface:

```python
class Environment:
    def list_jobs(self) -> list[JobPublic]: ...
    def inspect_job(self, job_id: str) -> JobPublic: ...
    def bid(self, job_id: str, price: Decimal) -> dict[str, Any]: ...
    def submit(self, job_id: str, artifact: str) -> VerifyResult: ...
    def check_balance(self) -> Decimal: ...
    def list_in_progress(self) -> list[str]: ...
    def end_tick(self) -> None: ...
    def terminated(self) -> bool: ...
    def finalize(self) -> EpisodeSummary: ...
```

v0.1 can use direct Python method calls rather than a JSON-RPC tool layer. The method names should still mirror the future tool API.

### 4.9 `solvent.harness.stub`

Provide deterministic harness policies for smoke testing.

Implement at least two stub modes:

1. `happy_path`: bids below the known generated acceptance range and submits a valid artifact.
2. `bad_delivery`: bids the same way but submits a malformed artifact.

The stub should not read hidden ground truth. It can construct a valid artifact from public inputs because this is the intended work product.

Expected API:

```python
class StubHarness:
    def __init__(self, mode: str): ...
    def run(self, env: Environment) -> None: ...   # drives tools only; never finalizes
```

Run strategy for `happy_path`:

1. list jobs
2. inspect first available job
3. bid a conservative fixed price, e.g. `$0.50`
4. if accepted, derive cleaned CSV from public input and submit it
5. continue calling `end_tick` (with a cheap `list_jobs`) each tick until termination; in v0.1 the stub completes at most one job and never re-bids a job

### 4.10 `solvent.cli.main`

Provide the first user-facing command.

Command:

```bash
solvent run --agent stub --seed 42
```

Useful options:

```bash
--stub-mode happy_path|bad_delivery
--start-balance 20.00
--horizon 3
--overhead 0.05
--tool-call-cost 0.01
--trace-path runs/seed-42-stub.jsonl
```

Terminal output should be concise:

```text
Solvent episode complete
seed: 42
agent: stub:happy_path
terminated: turn_cap
start_balance: 20.00
end_balance: 20.29
net: 0.29
trace: runs/seed-42-stub.jsonl
```

These numbers assume the documented config (`--horizon 3 --tool-call-cost 0.01 --overhead 0.05`, one job completed at tick 0): revenue `+0.50`, tool burn `6 × 0.01 = 0.06` (3× `list_jobs`, plus `inspect`/`bid`/`submit`), overhead `3 × 0.05 = 0.15` → net `+0.29`. Keep illustrative numbers consistent with the stated mechanics.

The CLI should derive `config_id` from the selected harness and mode, e.g. `stub:happy_path`, and pass it into `EnvConfig` so `EpisodeSummary.config_id` and terminal output share one source of truth.

Use `argparse` or `typer`. Prefer `argparse` if minimizing dependencies.

## 5. Episode Flow

The initial episode loop can be harness-driven:

```python
def run_episode(config: EnvConfig, harness: StubHarness) -> EpisodeSummary:
    env = Environment(config)
    try:
        harness.run(env)          # harness drives tools; it never finalizes
    finally:
        summary = env.finalize()  # only the env owner finalizes; idempotent
    return summary
```

The stub harness loop:

```python
completed_job = False
while not env.terminated():
    jobs = env.list_jobs()
    if jobs and not completed_job:
        job = env.inspect_job(jobs[0].id)
        bid = env.bid(job.id, Decimal("0.50"))   # one bid per job; bids are final
        if bid["accepted"]:
            artifact = build_clean_csv(job.inputs["csv"])
            env.submit(job.id, artifact)
            completed_job = True
    env.end_tick()
# harness.run returns None; run_episode() calls env.finalize()
```

This is intentionally simple. v0.2 can replace the stub with a ReAct harness that chooses one tool action per step.

## 6. Money Accounting Rules

Use these exact v0.1 rules:

1. `start_balance` is credited once at episode start.
2. Every environment tool call debits `tool_call_cost`.
3. `end_tick` debits `overhead_per_tick`, emits `overhead_charged`, then advances the clock.
4. A bid is accepted iff `price <= reservation_price`.
5. Each job accepts at most one bid; bids are final (no re-bidding). This prevents probing the hidden `reservation_price`.
6. Accepted bid price is stored as the job's contract price.
7. Revenue is credited only when the accepted job's artifact passes verification.
8. Failed verification does not credit revenue.
9. A passing job cannot be paid twice.
10. Termination is checked immediately after any debit/credit (insolvency: `balance <= 0`) and after `end_tick` (turn cap: `tick >= horizon_ticks`).

For v0.1, `est_cost` is not debited separately. Realized cost is represented by tool-call burn plus overhead. Keep `est_cost` on `Job` because v0.2 needs it for true value and omniscient/realizable reference policies.

The flat `tool_call_cost` is a v0.1 stand-in for the parent design's token-metered burn (`tokens × real per-token price`, TDD C.9). The metering *point* — the environment debits burn at the harness boundary — is the durable part and does not change; only the cost *formula* swaps to token-based when the LLM harness lands in v0.2.

## 7. Trace Schema

Each JSONL event should have this top-level shape:

```json
{
  "tick": 0,
  "kind": "bid_made",
  "payload": {
    "job_id": "dc-42-0",
    "price": "0.50"
  },
  "balance_after": "19.97",
  "burn_delta": "0.01"
}
```

Event payload guidance:

- Public observation events may include public job fields.
- Internal economic events may include accepted price and credited revenue.
- v0.1 traces should not include `reservation_price`, expected cleaned CSV, or rubric internals.
- Verifier results may include check names, pass/fail booleans, and messages.

Minimum happy-path trace:

```text
episode_started
board_seen          # burn_delta on this tool-call event
inspected           # burn_delta
bid_made            # burn_delta
bid_accepted        # resolution, burn_delta "0"
submitted           # burn_delta
verified_pass       # resolution, burn_delta "0"
paid                # resolution, burn_delta "0"
overhead_charged
tick_advanced
...
terminated
```

Burn representation is fixed for v0.1: `burn_delta` is the cost debited on that single event — tool-call burn on action events (`board_seen`, `inspected`, `bid_made`, `submitted`), the overhead amount on `overhead_charged`, and `"0"` on resolution/credit events (`bid_accepted`, `bid_declined`, `verified_pass`, `verified_fail`, `paid`, `terminated`). There is no standalone `burn_charged` event, and burn is never double-counted.

## 8. Error Handling

Define typed exceptions:

```python
class SolventError(Exception): ...
class UnknownJobError(SolventError): ...
class InvalidActionError(SolventError): ...
class AlreadyTerminatedError(SolventError): ...
```

Invalid actions should:

- charge burn if they are tool calls
- emit an `invalid_action` event
- return a structured error to the harness, or raise in tests

For v0.1, raising exceptions is fine for programming errors. For agent-facing invalid actions in v0.2, prefer structured tool errors.

## 11. v0.1 Acceptance Criteria

v0.1 is done when:

1. `solvent run --agent stub --seed 42` completes without errors.
2. A trace file is written in JSONL.
3. The trace contains a full episode from start to termination.
4. A valid `data_clean` submission is paid exactly once.
5. An invalid `data_clean` submission is not paid.
6. Tool-call burn and per-tick overhead affect the final balance.
7. The harness never sees `reservation_price`, `est_cost`, expected output, or rubric internals.
8. Re-running the same seed and config produces a byte-identical trace (v0.1 records no wall-clock timestamps).
9. Unit and integration tests pass.

## 13. v0.2 Handoff Notes

v0.1 should intentionally preserve extension points for v0.2:

- `reservation_price` already exists for pricing regret.
- `est_cost` exists for true-value and omniscient/realizable reference-policy calculations.
- `Market` can add decoys and more varied arrivals.
- The static board is intentionally narrow, but `arrival_tick` + ordered events leave room for v0.4+ expiry, reputation, and reactive market sampling.
