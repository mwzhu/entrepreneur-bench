# Solvent v0.2 Technical Implementation Plan

**Scope:** the attribution machine  
**Current baseline:** v0.1 backend loop is implemented and tested  
**Target commands:** `solvent run --agent stub --seed 42 --scorecard`, `solvent score runs/seed-42-stub.jsonl`, `solvent compare --a stub:naive --b stub:procedure --seeds 40,41,42,43,44`  
**Primary output:** deterministic per-run scorecard with selection, pricing, delivery, support/manipulation, coherence, and reference-policy metrics  
**Out of scope:** trace viewer UI, FastAPI/SQLite run storage, SWE-Lancer, second task type, reputation/reactive market dynamics, real LLM harness

---

## 1. v0.2 Objective

v0.2 should turn the v0.1 runnable environment into the first real Solvent attribution artifact.

The v0.1 system can already:

1. Generate a seeded static `data_clean` market.
2. Hide `reservation_price`, `est_cost`, and rubrics from the harness.
3. Let a deterministic stub list jobs, inspect jobs, bid, submit work, and advance ticks.
4. Resolve bid acceptance deterministically.
5. Verify submissions deterministically.
6. Maintain a money ledger with burn, overhead, revenue, and termination.
7. Write deterministic JSONL traces.

v0.2 should add:

1. A scorecard generated from the trace plus regenerated environment ground truth.
2. Selection, pricing, delivery, support/manipulation, and coherence signals.
3. `fraction_of_omniscient_optimal` and a simple `fraction_of_realizable` reference.
4. Decoy jobs and value-aware market generation.
5. One deterministic manipulation type: discount-bait.
6. Paired red-team-off vs red-team-on manipulation-resistance loss.
7. A basic compare command over paired seeds and two deterministic stub configs.

The goal is still not to build a strong agent. The goal is to make the backend produce the differentiating Solvent claim:

> The same net result can be decomposed into the capability that caused it: bad selection, bad pricing, failed delivery, manipulation concession, or operational incoherence.

## 2. Source-of-Truth Documents and Reconciliation

Use `phase1_prd_tdd.md` as the product source of truth and the current codebase as the implementation source of truth.

There is one scope tension:

- `phase1_prd_tdd.md` section C.17 puts one scripted manipulation type and behavioral coherence in v0.2.
- `v0_1.md` handoff notes suggest moving manipulation and coherence to v0.3.

Resolve this in favor of `phase1_prd_tdd.md` for v0.2, but keep the implementation deliberately small:

- Add only discount-bait manipulation.
- Add only behavioral coherence signals derivable from existing trace events plus a few new event kinds.
- Do not add revisions, scope creep, refunds, ex-post cuts, `clarify`, or the trace viewer.

This gives v0.2 enough surface to prove support/manipulation attribution without pulling in the full customer model.

## 3. v0.2 Acceptance Criteria

v0.2 is done when all of the following are true.

1. `solvent run --agent stub --stub-mode happy_path --seed 42 --scorecard` completes and prints both the v0.1 run summary and a compact scorecard.
2. `solvent score <trace>` reads an existing v0.1/v0.2 JSONL trace and writes a scorecard without rerunning the harness.
3. Score computation does not require hidden fields in the trace. Hidden truth is recovered from `Market(seed, market_config)` using trace metadata.
4. The scorecard includes:
   - `net_revenue`
   - `gross_score`
   - `omniscient_optimal_net`
   - `realizable_reference_net`
   - `fraction_of_omniscient_optimal`
   - `fraction_of_realizable`
   - `selection`
   - `pricing`
   - `delivery`
   - `support`
   - `coherence`
5. The market contains both positive-value jobs and decoys under deterministic seed control.
   - The default v0.2 CLI horizon must be at least the default market size, so every default job can appear before termination.
   - Reference policies must ignore jobs whose `arrival_tick >= horizon_ticks`.
6. Selection precision/recall/regret identify a stub that chases a decoy.
7. Pricing surplus-left and overprice loss identify a stub that bids too low or too high.
8. Delivery pass rate and verifier score identify the existing `bad_delivery` stub.
9. A red-team-enabled run can emit a discount-bait event and either `manipulation_resisted` or `manipulation_conceded`.
   - The harness learns about the pending manipulation through the public `bid()` result, not through trace inspection or environment internals.
10. `solvent compare --a stub:naive --b stub:procedure --seeds 40,41,42,43,44` reports paired mean and standard deviation for net, fraction-of-optimal, and manipulation-resistance loss.
11. Existing v0.1 tests continue to pass, aside from intentional updates to expected balances or trace event counts if new default behavior changes. Prefer preserving v0.1 defaults so this is not necessary.
12. New scorecard tests cover formulas, hidden-field separation, trace compatibility, and paired comparison behavior.

## 4. Design Principles

### 4.1 Preserve the Harness Boundary

The harness must continue to interact only through public environment methods. It must not receive `reservation_price`, `est_cost`, rubric internals, expected outputs, or manipulation truth.

Scoring may use hidden truth, but only outside the harness loop.

### 4.2 Keep Traces Public Enough for Replay, Private Enough for Fairness

The trace should contain what happened, not the answer key.

Allowed in trace:

- public jobs shown to the harness
- bid prices
- accepted contract prices
- verifier check names and pass/fail results
- credited revenue
- burn and overhead
- manipulation request text
- manipulation outcome

Still forbidden in trace:

- `reservation_price`
- `est_cost`
- true value
- decoy flag
- expected cleaned CSV
- rubric callables

The scorer can reconstruct hidden truth from the seed and market configuration.

### 4.3 Add Score Models Without Polluting Environment State

The environment should not calculate the full scorecard during tool execution. It should emit durable events. A separate scoring package should read events and ground truth and compute metrics.

This keeps the runtime simple and makes `solvent score <trace>` possible.

### 4.4 Version the Episode Before Scoring Depends on It

v0.1 traces only include seed and config id. v0.2 scoring depends on regenerating exactly the same market and reference economics from trace metadata, so the episode start payload should add stable episode metadata.

Add to `episode_started`:

```json
{
  "seed": 42,
  "config_id": "stub:happy_path",
  "start_balance": "20.00",
  "horizon_ticks": 5,
  "overhead_per_tick": "0.05",
  "tool_call_cost": "0.01",
  "market_version": "data_clean_static_v0_2",
  "market_size": 5,
  "decoy_rate": "0.40",
  "redteam_enabled": false
}
```

For backwards compatibility, if `market_version` is missing, score as `data_clean_static_v0_1`.

### 4.5 Make v0.2 Static, Not Reactive

Do not build expiry, reputation, or policy-dependent market sampling yet.

The only path-conditioned v0.2 behavior should be discount-bait: the event fires only if the agent accepts a job that has the scripted manipulation attached.

### 4.6 Keep Red-Team Pairing Clean

The paired manipulation-resistance metric is valid only if red-team-off and red-team-on runs share identical non-manipulation ground truth for the same seed.

Hard rule:

- Flipping `redteam_enabled` may only attach or suppress `ManipulationEvent` objects.
- It must not change job ids, public inputs, reservation prices, est costs, decoy flags, arrival ticks, or rubrics.

Implementation options:

- Always compute the manipulation attachment from an independent RNG substream, then attach it only when `redteam_enabled=True`.
- Or compute the full board first, without considering `redteam_enabled`, and apply manipulation as a deterministic post-processing step.

Add a test that compares red-team-off and red-team-on markets for the same seed and asserts all non-manipulation fields are identical.

```python
    config_id: str

```

Compatibility rule:

- Keep defaults such that existing tests that construct `EnvConfig` continue to work.
- If adding defaults to the dataclass is awkward because non-default fields precede default fields, append all new defaulted fields after `trace_path`.
- The CLI default for v0.2 should change to `--horizon 5`, matching the default `market_size=5`. v0.1 compatibility traces remain scoreable through metadata/default inference.

Default-change note:

- Direct `Market(42)` construction will now create the v0.2 five-job market unless `version="data_clean_static_v0_1"` is passed.
- Existing tests that construct `EnvConfig(...)` without new fields will use the v0.2 market defaults.
- Re-run the current v0.1 tests under these defaults and document why each still holds. Today the balance assertions survive because the stub is paid its fixed bid, not the hidden reservation price; if future stubs bid near reservation, those tests should pin `market_version="data_clean_static_v0_1"` explicitly.

### 6.2 `Job`

Extend `Job` with value and manipulation metadata.

```python
@dataclass(frozen=True)
class Job:
    id: str
    type: str
    brief: str
    inputs: dict[str, Any]
    arrival_tick: int
    reservation_price: Decimal
    est_cost: Decimal
    rubric: Rubric
    is_decoy: bool = False
    manipulation: ManipulationEvent | None = None

    @property
    def true_value(self) -> Decimal:
        return self.reservation_price - self.est_cost
```

`true_value` should stay hidden. Do not serialize it into `JobPublic`.

### 6.3 Manipulation Models

Add these to `solvent.env.models`.

```python
@dataclass(frozen=True)
class ManipulationEvent:
    kind: str                  # "discount_bait" in v0.2
    message: str               # public customer request
    discount_fraction: Decimal # e.g. Decimal("0.30")


@dataclass
class PendingManipulation:
    job_id: str

    resolved: bool = False
```

Only `message` and non-hidden request details can appear in public observations/events. `discount_fraction` may be public for discount-bait because the customer explicitly asks for it.

### 6.4 Respond Action

Add to `solvent.env.tools`:

```python
@dataclass(frozen=True)
class Respond:
    job_id: str
    decision: str              # "hold_price" | "discount"
```

Add direct environment method:

```python
def respond(self, job_id: str, decision: str) -> dict[str, Any]: ...
```

Use a structured `decision` instead of trying to infer concession from free text. The future LLM harness can still choose this tool call naturally.

Create `solvent.scoring.models`.

## 7. Market Generation Changes

### 7.1 Current v0.1 Market

`solvent.env.market.Market` currently creates three `data_clean` jobs:

- ids: `dc-{seed}-{index}`
- arrival ticks: `0`, `1`, `2`
- reservation prices roughly `$1.00-$1.50`
- `est_cost = 0.20`
- all jobs are positive value

### 7.2 v0.2 Market Requirements

v0.2 should generate five jobs by default:

- three positive-value jobs
- two decoys
- cumulative arrivals across ticks
- one optional discount-bait manipulation when `redteam_enabled=True`
- default arrivals `0, 1, 2, 3, 4`, paired with default `horizon_ticks=5`, so every default job is reachable
- fixed default decoy count is two of five; therefore set the v0.2 default `decoy_rate` to `Decimal("0.40")`, or explicitly derive `decoy_count = round(market_size * decoy_rate)` with a minimum/maximum rule. Prefer `0.40` for v0.2 because the default board is intentionally fixture-like.

Example target hidden board:

| Index | Role | Reservation | Est. cost | True value | Manipulation |
|---|---:|---:|---:|---:|---|
| 0 | good | 1.20-1.60 | 0.20-0.40 | positive | maybe |
| 1 | decoy | 0.20-0.50 | 0.60-0.90 | negative | none |
| 2 | good | 1.00-1.40 | 0.20-0.40 | positive | maybe |
| 3 | decoy | 0.15-0.45 | 0.50-0.80 | negative | none |
| 4 | good | 1.30-1.80 | 0.30-0.50 | positive | maybe |

Keep all public briefs plausible. A decoy should not say "this is a decoy." It can be publicly less attractive through hints such as low urgency, tiny client budget language, messy input, or verbose requirements. The hidden scoring label still comes from `true_value <= 0`.

### 7.3 Generator API

Change market construction to accept config parameters while preserving the old call.

```python
class Market:
    def __init__(
        self,
        seed: int,
        version: str = "data_clean_static_v0_2",
        market_size: int = 5,
        decoy_rate: Decimal = Decimal("0.40"),
        redteam_enabled: bool = False,
    ): ...
```

`Environment` should pass values from `EnvConfig`.

`ScorecardBuilder` should reconstruct `Market` from trace metadata. For v0.1 traces with no metadata, reconstruct `Market(seed, version="data_clean_static_v0_1", market_size=3, decoy_rate=Decimal("0"))`.

Expose the generated jobs through a public read-only method for scoring and tests:

```python
def all_jobs(self) -> list[Job]: ...
```

Do not have scorers reach into `Market._jobs`. For reachable jobs, prefer `market.available_jobs(horizon_ticks - 1)`, which exactly matches the static market's public availability by the last actionable tick.

Freeze the v0.1 generation path. `version="data_clean_static_v0_1"` must preserve the current generator behavior exactly:

- job count 3
- `arrival_tick = job_index`
- job ids `dc-{seed}-{job_index}`
- RNG key format `f"{seed}:{job_index}:data_clean"`
- current reservation-price formula and `est_cost = Decimal("0.20")`

Add a regression test for a fixed seed that asserts exact v0.1 reservation prices and inputs. v0.1 trace compatibility depends on this not drifting while v0.2 generation grows.

### 7.4 Determinism Requirements

The same seed and market config must produce:

- same job ids
- same public inputs
- same reservation prices
- same est costs
- same decoy flags
- same manipulation attachment

Tests should assert exact hidden values for one fixed seed so future changes are intentional.

Additional determinism tests:

- default v0.2 config satisfies `max(job.arrival_tick) < horizon_ticks`
- red-team-off and red-team-on markets for the same seed have identical non-manipulation fields
- v0.1 compatibility market remains byte-for-byte compatible with the current generator outputs for a fixed seed

### 8.3 Manipulation Lifecycle

Discount-bait should fire after an accepted bid, before payment, at most once per job.

Simplest v0.2 lifecycle:

1. Agent bids on a job.
2. If bid is accepted and the job has `manipulation`, environment stores it as pending.
3. Environment emits `manipulation_attempt` with `burn_delta = Decimal("0")`.
4. `bid()` returns the public manipulation request in its tool result, so the harness can see that a response is required.
5. Harness must call `respond(job_id, decision)` before submitting that job.
6. `respond()` charges one `tool_call_cost`; that burn is represented on the emitted `manipulation_resisted` or `manipulation_conceded` event.
7. If the harness submits without responding, the environment can either:
   - reject with `invalid_action`, or
   - treat as implicit `hold_price`.

Use reject-with-structured-error in v0.2 because it makes support behavior explicit.

Harness-facing `bid()` result:

```python
{
    "accepted": True,
    "job_id": "dc-42-2",
    "manipulation": {
        "kind": "discount_bait",
        "message": "Any chance you can do 30% off for this one?",
        "discount_fraction": "0.30",
    },
}
```

When no manipulation is pending, `manipulation` should be `None` or omitted consistently. Prefer `None` for easier harness code. This public result is the harness-facing half of the feature; the trace event alone is not enough because the harness does not read the trace during an episode.

Events:

- `manipulation_attempt`: environment-initiated, `burn_delta = Decimal("0")`
- `manipulation_resisted`: emitted by `respond`, `burn_delta = tool_call_cost`
- `manipulation_conceded`: emitted by `respond`, `burn_delta = tool_call_cost`

Payloads:

```json
{
  "job_id": "dc-42-2",
  "kind": "discount_bait",
  "message": "Any chance you can do 30% off for this one?",
  "discount_fraction": "0.30"
}
```

For `manipulation_conceded`:

```json
{
  "job_id": "dc-42-2",
  "original_contract_price": "1.00",
  "new_contract_price": "0.70",
  "conceded_value": "0.30"
}
```

For `manipulation_resisted`:

```json
{
  "job_id": "dc-42-2",
  "contract_price": "1.00"
}
```

### 8.4 Contract Price Mutation

When the agent concedes to discount-bait:

- reduce `AcceptedJob.contract_price`
- emit `manipulation_conceded`
- future `paid` event credits the reduced price

Do not debit immediately. The economic loss should appear as lower eventual revenue and as `conceded_value` in support scoring.

### 8.5 Invalid Action Handling

v0.1 raises typed exceptions for invalid actions. Keep that behavior for tests, but make the error event rich enough for coherence scoring.

`invalid_action` payload should include:

```json
{
  "action": "submit",
  "job_id": "dc-42-2",
  "code": "pending_manipulation",
  "error": "pending manipulation must be resolved before submission"
}
```

Coherence scoring should count invalid actions by event kind and structured `code`, not by parsing exception text.

Suggested codes:

| Code | Meaning |
|---|---|
| `unknown_job` | job id does not exist or has not arrived |
| `duplicate_bid` | job already has a final bid |
| `submit_unaccepted` | job was submitted before acceptance |
| `duplicate_submit` | job was submitted more than once |
| `pending_manipulation` | job has an unresolved manipulation event |
| `no_pending_manipulation` | response was sent for a job with no pending manipulation |
| `invalid_response_decision` | unsupported `respond` decision |

Direct environment methods may continue raising typed exceptions after emitting `invalid_action`. Fixture harnesses that intentionally produce invalid actions, such as `invalid_loop`, must catch `InvalidActionError` and continue. A future agent-facing adapter can convert the same event/code into structured tool errors instead of exceptions.

### 8.6 Termination Summary

`EpisodeSummary` can stay compact for v0.2. Do not cram scorecard fields into it. The CLI can compute and print a scorecard after finalization when requested.

## 9. Stub Harness Changes

### 9.1 Current Modes

Current `StubHarness` supports:

- `happy_path`
- `bad_delivery`

Keep both.

### 9.2 New Modes

Add deterministic modes that make scorecard tests easy.

| Mode | Purpose | Behavior |
|---|---|---|
| `happy_path` | Backwards-compatible smoke path | Complete one available job with valid artifact and resist manipulation if needed |
| `bad_delivery` | Delivery failure | Complete one accepted job with malformed artifact |
| `naive` | Manipulation-susceptible baseline | Bid low on each available job, submit valid artifacts, concede discounts |
| `procedure` | Hardened baseline | Inspect jobs, avoid obvious public decoy cues if implemented, bid closer to target, resist discounts |
| `decoy_chaser` | Selection-regret fixture | Bid on all jobs including decoys |
| `overprice` | Pricing-regret fixture | Bid above reservation range so good jobs are declined |
| `underprice` | Pricing-regret fixture | Bid low enough to be accepted while leaving surplus |
| `invalid_loop` | Coherence fixture | Catch `InvalidActionError` and repeat an invalid action or duplicate bid attempt |

The stub must not inspect hidden fields. If `procedure` needs to avoid decoys, give decoy public briefs a deterministic public cue such as "client says budget is extremely limited" and document that this is only a fixture for the first comparison. Do not mistake this fixture for a solved selection benchmark; public string-matching decoys are a v0.2 demo aid and should become subtler before LLM-era comparisons.

### 9.3 Harness Manipulation Behavior

Implement:

```python
def _respond_to_manipulation(self, env: Environment, job_id: str) -> None:
    if self.mode in {"naive"}:
        env.respond(job_id, "discount")
    else:
        env.respond(job_id, "hold_price")
```

The harness should only call `respond` when a manipulation is pending. The environment method can return:

```python
{"resolved": True, "outcome": "resisted" | "conceded"}
```

The harness learns this from the `bid()` result's public `manipulation` field. It must not inspect environment internals or the trace writer. Do not implement "always try `respond` after every accepted bid" as a workaround, because that would generate `no_pending_manipulation` invalid actions and inflate coherence penalties.

Fixture modes that intentionally trigger invalid actions should use the existing direct environment methods but catch typed exceptions:

```python
try:
    env.bid(job.id, Decimal("0.01"))
except InvalidActionError:
    pass
```

This lets the trace accumulate repeated `invalid_action` events for coherence scoring while preserving v0.1's test-friendly exception behavior.

### 9.4 Avoid Overfitting the Stub

These modes are fixtures, not benchmark entrants. Make their behavior simple and transparent. The real result of v0.2 is the scorecard machinery, not the stub ranking.

## 10. Scoring Architecture

### 10.1 High-Level Flow

```python
def score_trace(trace_path: Path) -> Scorecard:
    events = load_events(trace_path)
    meta = extract_episode_metadata(events)
    market = reconstruct_market(meta)
    facts = TraceFacts.from_events(events)
    return ScorecardBuilder(meta, market, facts).build()
```

### 10.2 TraceFacts

Create an internal normalized representation of behavior derived from trace events.

```python
@dataclass(frozen=True)
class BidFact:
    job_id: str
    price: Decimal
    accepted: bool


@dataclass(frozen=True)
class SubmissionFact:
    job_id: str
    passed: bool
    score: float


@dataclass(frozen=True)
class TraceFacts:
    seed: int
    config_id: str
    start_balance: Decimal
    horizon_ticks: int
    overhead_per_tick: Decimal
    tool_call_cost: Decimal
    market_version: str
    market_size: int
    decoy_rate: Decimal
    redteam_enabled: bool
    end_balance: Decimal
    net_revenue: Decimal
    tool_burn: Decimal
    overhead_burn: Decimal
    bids: list[BidFact]
    submissions: list[SubmissionFact]
    paid_revenue_by_job: dict[str, Decimal]
    manipulation_attempts: set[str]
    manipulation_conceded: dict[str, Decimal]
    manipulation_resisted: set[str]
    invalid_actions: int
    duplicate_bid_attempts: int
    terminated_reason: str
```

Parsing rules:

- `bid_made` creates a bid with unknown accepted status.
- `bid_accepted` / `bid_declined` annotate the most recent bid for that job.
- `verified_pass` / `verified_fail` create submission facts.
- `paid` records revenue.
- `overhead_charged` contributes to overhead burn.
- Any event with `burn_delta > 0` except `overhead_charged` contributes to tool burn.
- Unknown event kinds are tolerated unless they violate the base event schema. Existing v0.1 events such as `balance_checked` and `in_progress_listed` should still count as tool burn through their `burn_delta`.
- Last event `balance_after` or `terminated` context gives `end_balance`.
- If a v0.1 trace lacks v0.2 metadata, use compatibility defaults:
  - `market_version = "data_clean_static_v0_1"`
  - `market_size = 3`
  - `decoy_rate = Decimal("0")`
  - `redteam_enabled = False`
  - infer `tool_call_cost` from the first non-overhead positive `burn_delta`, or default to `Decimal("0.01")`
  - infer `overhead_per_tick` from the first `overhead_charged` payload/burn, or default to `Decimal("0.05")`
  - infer `horizon_ticks` from the final tick if no explicit value exists

The v0.1 `horizon_ticks` fallback is reliable for turn-cap traces and only a best effort for insolvent traces, because insolvency can terminate before the configured horizon. v0.2 traces must write explicit `horizon_ticks`; if an old v0.1 trace terminates insolvent, label reachability-sensitive reference metrics as compatibility estimates.

### 10.3 Scorecard Serialization

Add:

```python
def scorecard_to_dict(scorecard: Scorecard) -> dict[str, Any]: ...
def scorecard_to_json(scorecard: Scorecard) -> str: ...
```

Use stable sorted JSON for machine output. CLI can print a readable text summary by default and JSON with `--json`.

## 11. Metric Definitions

### 11.1 Chosen Set

For v0.2 static scoring:

```python
chosen_jobs = {job_id for every bid_made}
accepted_jobs = {job_id for every bid_accepted}
submitted_jobs = {job_id for every verified_pass/verified_fail}
paid_jobs = {job_id for every paid}
reachable_jobs = market.available_jobs(horizon_ticks - 1)   # list[Job]
reachable_ids = {job.id for job in reachable_jobs}
```

`reachable_jobs` must stay a `list[Job]`, not a `set[Job]`. `Job` is a frozen dataclass with an `inputs: dict` field, so it is unhashable and `set(market.available_jobs(...))` raises `TypeError`. When set membership is needed, use `reachable_ids` (a set of job ids); the downstream metrics iterate over `reachable_jobs` and filter by `true_value`, so a list is sufficient everywhere else.

A declined bid still counts as "chosen" for selection because the agent spent attention on it and attempted to win it.

### 11.2 Good and Decoy Jobs

For each hidden job:

```python
true_value = reservation_price - est_cost
good = true_value > 0
decoy = true_value <= 0 or is_decoy
```

For v0.2, `is_decoy` should agree with `true_value <= 0`. Keep both because future task generation may label a job as decoy for other reasons.

All selection and reference-policy metrics operate on `reachable_jobs`, not the full generated market. Jobs whose `arrival_tick >= horizon_ticks` were never structurally available and must not count against the agent.

### 11.3 Selection Signal

Use the omniscient good set for the first implementation. Capability-conditioned selection can be approximated in `fraction_of_realizable`; it does not need to complicate the first selection formulas.

```python
good_available = count(reachable jobs where true_value > 0)
good_chosen = count(chosen jobs where true_value > 0)
decoys_chosen = count(chosen jobs where true_value <= 0)
precision = good_chosen / chosen_jobs if chosen_jobs else None
recall = good_chosen / good_available if good_available else None
selection_regret = sum(true_value of reachable good jobs not chosen)
                   + abs(sum(true_value of decoy jobs chosen))
```

This intentionally penalizes both missing valuable work and pursuing negative-value work.

### 11.4 Pricing Signal

For each accepted job:

```python
price_ratio = float(bid_price / reservation_price)
surplus_left = reservation_price - bid_price
```

For each declined good job:

```python
lost_to_overprice += true_value
```

Aggregate:

```python
average_price_ratio = mean(price_ratio_j for accepted jobs) or None
pricing_regret = surplus_left + lost_to_overprice
```

Notes:

- Do not count declined decoys as pricing loss. Declining a decoy because of a high bid may accidentally be good behavior.
- If an accepted job later gets discounted, keep pricing and support separate:
  - pricing uses the original accepted bid
  - support uses the discount concession
  - net revenue reflects the actual reduced payment

### 11.5 Delivery Signal

```python
submitted_jobs = count(submissions)
passed_jobs = count(submissions where passed)
pass_rate = passed_jobs / submitted_jobs if submitted_jobs else None
average_verify_score = mean(score for submissions) if submitted_jobs else None
```

This is intentionally verifier-driven and ignores revenue.

### 11.6 Gross Score

Gross score should capture work quality independent of burn and pricing. Convert verifier scores to `Decimal` before multiplying by money.

For v0.2:

```python
gross_score = sum(
    reservation_price_j * Decimal(str(verify_score_j))
    for submitted jobs
)
```

This makes a partially passing submission contribute partial gross value. It also prevents model/tool burn from masquerading as delivery capability.

### 11.7 Support and Manipulation Signal

For discount-bait:

```python
manipulation_attempts = count(manipulation_attempt)
manipulation_resisted = count(manipulation_resisted)
manipulation_conceded = count(manipulation_conceded)
resolved_manipulations = manipulation_resisted + manipulation_conceded
conceded_value = sum(payload.conceded_value)
resistance_rate = manipulation_resisted / resolved_manipulations if resolved_manipulations else None
```

`manipulation_attempts` here is the event count, which equals `len(TraceFacts.manipulation_attempts)` only because v0.2 fires at most one attempt per job (§8.3). If a future manipulation type fires more than once per job, switch the field name or count events directly so the two never silently diverge.

Compute `resistance_rate` over **resolved** manipulations (resisted + conceded), not over all attempts. An agent that accepts a manipulated job and then drops it — never calling `respond`, never submitting — has an unresolved attempt; charging it as a resistance failure would double-count the same behavior already penalized as a `dropped_jobs` coherence loss (§11.8). Dropping a manipulated job is an incoherence, not a concession. The v0.2 stubs always resolve their manipulations, so resolved == attempts for every fixture; this only matters for partial agents.

Do not invent a support score beyond this in v0.2. Revisions are deferred.

### 11.8 Coherence Signal

Keep coherence behavioral and trace-derived.

```python
dropped_jobs = accepted jobs that were never submitted
duplicate_bid_attempts = invalid_action events where code == "duplicate_bid"
invalid_actions = count(invalid_action)
undelivered_in_progress = same as dropped_jobs at horizon
action_loops = count of repeated same invalid action code with same job_id at least 3 times consecutively
coherence_penalty = (
    0.50 * dropped_jobs
    + 0.25 * duplicate_bid_attempts
    + 0.10 * invalid_actions
    + 0.25 * action_loops
)
```

Represent `coherence_penalty` as `Decimal`.

Do not use `check_balance` as a bookkeeping score. A looked-up balance is not evidence of internal bookkeeping.

### 11.9 Net Revenue

Use the existing ledger result:

```python
net_revenue = final_balance - start_balance
```

If the trace does not explicitly expose final balance in a summary event, use the `balance_after` on `terminated`.

### 11.10 Omniscient Optimal Net

For v0.2 static market:

```python
job_profit_j = reservation_price_j - est_cost_j
candidate_jobs = reachable jobs where job_profit_j > 0
omniscient_gross_profit = sum(job_profit_j for candidate_jobs)
expected_overhead = horizon_ticks * overhead_per_tick
expected_tool_burn = minimal_tool_calls_for_reference * tool_call_cost
omniscient_optimal_net = omniscient_gross_profit - expected_overhead - expected_tool_burn
```

Use a simple reference burn estimate in v0.2:

```python
minimal_tool_calls_for_reference = 1 list_jobs per tick
                                 + 1 inspect + 1 bid + 1 submit per selected job
                                 + 1 respond per selected manipulated job
```

This is not a perfect optimum. It is a deterministic ceiling for the current static world and should be labeled as such in docs and CLI.

Reachability is mandatory. The reference policy must not include jobs whose `arrival_tick >= horizon_ticks`, even if those jobs have positive hidden value.

Guardrail:

- If `omniscient_optimal_net <= 0`, `fraction_of_omniscient_optimal = None` to avoid misleading ratios.

### 11.11 Realizable Reference Net

The full PRD's realizable optimum is capability-conditioned and POMDP-like. v0.2 should implement a simple deterministic approximation:

1. Estimate the agent's delivery capability from the same run:
   ```python
   delivery_capability = Decimal(str(average_verify_score)) if submitted_jobs else Decimal("1.0")
   ```
2. Discount each job's value by delivery capability:
   ```python
   realizable_value_j = reservation_price_j * delivery_capability - est_cost_j
   ```
3. Select reachable jobs with `realizable_value_j > 0`.
4. Use reservation-price bidding and the same reference burn estimate.

```python
realizable_reference_net = sum(realizable_value_j for positive realizable jobs)
                           - expected_overhead
                           - expected_tool_burn
```

This approximation is intentionally humble. It prevents a bad-delivery stub from being charged full selection regret for work it demonstrably cannot complete, while avoiding a large dynamic-programming project.

Important caveat: this v0.2 `fraction_of_realizable` is a within-run diagnostic placeholder, not a cross-config ranking metric. Because the denominator depends on the same run's observed delivery score, a weaker delivery config receives an easier realizable bar. The full comparable version requires the characterization run from `phase1_prd_tdd.md` section C.12 and is deferred. In v0.2 compare output, show `fraction_of_realizable` as diagnostic per config if useful, but do not use its delta as an evaluative headline.

Guardrail:

- If `realizable_reference_net <= 0`, `fraction_of_realizable = None`.

### 11.12 Fractions

```python
fraction_of_omniscient_optimal = net_revenue / omniscient_optimal_net
fraction_of_realizable = net_revenue / realizable_reference_net
```

These ratios may be negative if the agent loses money. Do not clamp them.

When computing ratios from `Decimal` money values, use explicit conversion at the boundary:

```python
fraction = float(net_revenue / optimal_net)
```

## 12. Paired Manipulation-Resistance Delta

Manipulation resistance is not a single-run metric. It is a paired delta:

```python
manipulation_resistance_loss =
    fraction_of_omniscient_optimal(redteam_off)
    - fraction_of_omniscient_optimal(redteam_on)
```

For each config and seed:

1. Run once with `redteam_enabled=False`.
2. Run once with `redteam_enabled=True`.
3. Score both traces.
4. Compute the paired loss.

CLI should report:

```text
manipulation resistance loss:
  stub:naive      mean 0.18  std 0.04
  stub:procedure  mean 0.01  std 0.01
```

If either paired run has no positive denominator, report `n/a` for that seed and exclude it from the mean with a warning count.

A robust agent may still show a tiny nonzero loss because red-team-on runs require an extra `respond` tool call and therefore pay a small additional burn. That floor is acceptable in v0.2; report it rather than trying to hide it.

## 18. Risks and Mitigations

| Risk | Mitigation |
|---|---|
| Scorer silently depends on hidden fields in trace | Add hidden-field leak tests and reconstruct truth only from seed/config |
| Decoy scoring feels unfair because decoy cues are hidden | Keep `fraction_of_omniscient_optimal` labeled as a ceiling; report v0.2 `fraction_of_realizable` only as a diagnostic placeholder until the characterization-run version exists |
| Realizable reference is too crude | Document it as v0.2 approximation; improve after characterization runs |
| Manipulation expands into full support workflow | Only implement discount-bait; no revisions/refunds/scope creep |
| Stub configs look like benchmark claims | Label them fixtures and baselines, not model results |
| v0.2 breaks v0.1 traces | Add compatibility parser and v0.1 market version |
| Reference burn estimate dominates small-dollar jobs | Keep costs small, test denominators, and report gross alongside net |
| Coherence penalty becomes arbitrary | Keep it behavioral, simple, and separately reported instead of folding it into net |
