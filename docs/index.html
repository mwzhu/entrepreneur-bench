01 · Why

A final profit number is an answer key with the work erased.

Vending-Bench scored agents on net worth alone. Two agents can hit the same balance for opposite reasons — and that single number can't tell them apart.

  • The inspiration: running a business tests long-horizon coherence — the skill frontier models still lack.
  • The gap: net worth says how much, never why. Real-world tests (Project Vend) are slow and noisy.
  • Our move: we own the environment, so we can compute the perfect-foresight optimum.
  • The payoff: the gap to optimum = regret, split across the skills that caused it.

We score six skills independently — each against its own optimum:

Job selection

Takes the profitable jobs, skips the decoy traps.

Pricing

Bids near the client's hidden ceiling, not below it.

Delivery

The work actually passes the client's rubric.

Tool selection

Hires the cheapest model good enough for the job.

Coherence

No dropped work, re-bids, or action loops.

Support

Holds the line when a client pushes for a discount.

02 · The world

A freelance business in a sealed, replayable world.

Open the job board → accept or haggle → deliver → get paid on success. A daily overhead, the delivery fee, and the agent's own tokens all drain the balance. Game over at the horizon or when it goes broke.

The one twist that makes pricing hard: each job has a hidden ceiling, and the agent gets one shot at it.

03 · Jobs & market

The agent sees the work. We see the answer key.

Jobs are small auto-gradable gigs (clean a CSV, extract a field). Each has a public face the agent decides on, and a hidden record we grade against.

04 · The agent

A deliberately plain agent — we measure the model, not our scaffolding.

A basic ReAct loop: read the situation, call one tool, repeat. One system prompt, a persistent notebook, and 18 tools. No hints about decoys, ceilings, or which model fits which job — it has to figure the market out.

"You run a tiny freelance services business… maximize your final balance. Each job shows a visible starting_price and a hidden ceiling… you get a single delivery attempt… The episode terminates when the horizon is reached or the business becomes insolvent." — the system prompt, in brief

05 · Scoring

Replay each run against the best it could have done.

Because we set every value, we can compute reference policies and grade the agent as a fraction of each — then read where the gap comes from.

  • Omniscient optimum — perfect foresight, every delivery passes. The hard ceiling.
  • Realizable — same plan, discounted by real delivery luck. The achievable ceiling.
  • Threshold policy — a simple greedy baseline. A good agent should beat it.
  • Headline = fraction of omniscient optimum; the rest is regret.
  • Regret is split per skill — selection, pricing, delivery, tool, coherence, support.
  • Luck removed via an expected-value control variate (price × true pass-prob).
One honest caveat Per-skill regrets are each measured against their own optimum, so they rank where loss concentrates — they don't add up to the total gap. Early-terminated runs are graded only over reachable jobs.

07 · Limitations & next

The toy world is the easy part to fix. The decomposition is the idea worth keeping.

  • Real tasks — ground gigs in something like SWE-Lancer.
  • Real delivery models — swap synthetic tools for actual models.
  • Measured pass rates — characterize them, don't hand-author.
  • Competition — rival agents, arena, price wars.
  • Parallel jobs — let it run a portfolio, not one at a time.
  • Richer social layer — negotiation, haggling, red-team pressure.
  • Reputation that gates access to better work.
  • Longer horizons, more models, repeated seeds.
