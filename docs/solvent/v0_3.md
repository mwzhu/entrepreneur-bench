# Solvent v0.3 Technical Implementation Plan

**Scope:** the demo layer on top of the v0.2 attribution backend  
**Current baseline:** v0.2 scoring, paired comparison, manipulation pairing, deterministic JSONL traces, and stub modes are implemented and tested  
**Target commands:** `solvent demo`, `solvent compare --a stub:naive --b stub:procedure --seeds 40,41,42,43,44 --redteam-paired --viewer`, `solvent view runs/demo-v0_3`  
**Primary output:** a reproducible paired-seed demo directory containing traces, scorecards, a comparison manifest, and a clickable trace viewer with a clear dollar delta and per-stage attribution  
**Out of scope:** second task type, characterization command, dev/test seed split, reputation/reactive market dynamics, SWE-Lancer, real LLM harness, persistent multi-user run storage

---

## 1. v0.3 Objective

v0.3 should turn the v0.2 backend into the first demoable Solvent artifact.

The v0.2 system can already:

1. Run a seeded static `data_clean` freelance episode.
2. Compare two deterministic stub configs over paired seeds.
3. Emit JSONL traces and scorecards.
4. Compute selection, pricing, delivery, support/manipulation, and coherence signals.
5. Measure red-team-off vs. red-team-on manipulation-resistance loss.

v0.3 should add:

1. A one-command demo run for `stub:naive` vs. `stub:procedure` over a fixed seed set.
2. A durable run manifest that links each config, seed, red-team condition, trace, and scorecard.
3. A comparison scoreboard that foregrounds net revenue, fraction of optimal, pricing regret, selection regret, delivery, and manipulation-resistance loss.
4. Balance-over-time data for each trace.
5. A clickable trace viewer:
   - event timeline on the left
   - event detail / artifact / verifier verdict in the center
   - scorecard and balance curve on the right
   - comparison tab for the paired-seed scoreboard
6. Enough trace detail to make the "money shot" visible: naive concedes the discount, procedure resists, and the delta appears in the ledger and scorecard.

The goal is not to make the environment more realistic yet. The goal is to make the existing attribution machine legible and compelling.

## 2. Source-of-Truth Reconciliation

Use `phase1_prd_tdd.md` as the product source of truth and the current codebase as the implementation source of truth.

The PRD says v0.3 is:

> Two configs (`naive` vs `+procedure`); paired-seed scoreboard with a dollar delta; trace viewer, built only after the backend scorecard is correct and interpretable.

The current code already has the first half:

- `solvent.cli.main._compare`
- `solvent.scoring.compare`
- `StubHarness("naive")`
- `StubHarness("procedure")`
- per-run scorecards via `score_trace`
- `summary.json` in compare output

Therefore v0.3 should avoid broad backend rewrites. The work is mostly:

- enriching compare output into a full demo manifest
- adding trace-derived view models
- adding a static/local viewer
- adding a convenience command that generates the canonical demo
- adding tests that prevent the viewer artifacts from drifting away from the scorecard truth

### 2.1 Static Viewer First, API Seam Later

`phase1_prd_tdd.md` names FastAPI, SQLite, and a React/HTML viewer as the eventual stack. The current package has zero runtime dependencies and no storage layer. For v0.3, prefer a static viewer generated into the run directory:

- No server required to inspect the artifact.
- No dependency jump before the demo.
- The generated JSON files are easy to diff and test.
- The same view-model functions can later back FastAPI/SQLite without changing scoring.

The browser should load a generated `viewer/data.js` file, not fetch JSON at runtime:

```javascript
window.SOLVENT_DATA = { /* manifest, summary, and per-trace view models */ };
```

This keeps the demo robust under `file://` and avoids local browser fetch restrictions. Still write the canonical `.json` files too; those JSON files remain the tested, diffable seam for a future FastAPI/SQLite backend. `data.js` is only a wrapper around the same Python-generated dictionaries, not a second data path.

Add a tiny `solvent view` command that prints the static entry path and can optionally serve the directory with Python's standard library. Defer persistent SQLite indexing and FastAPI endpoints until there is more than one demo directory worth managing.

## 4. Demo Product Shape

### 4.1 Primary Demo Narrative

The default demo should make this story visible without narration:

1. Both configs face the same hidden markets on the same seeds.
2. The baseline scoreboard shows `Net revenue (baseline, red-team off)`, where the dollar delta is driven by pricing plus selection.
3. `stub:naive` accepts discount-bait and gives away revenue in red-team-on traces.
4. `stub:procedure` resists manipulation and avoids public decoy jobs.
5. The separate `Manipulation-resistance loss (red-team on minus off)` row isolates the manipulation stressor.
6. The per-stage scorecard localizes the difference to pricing/support and selection, not delivery.
7. Clicking the manipulation event shows:
   - the public customer request
   - whether the harness conceded or resisted
   - changed contract price for concession
   - corresponding paid amount

### 4.2 Suggested Default Seeds

Use the PRD's minimum of five paired seeds:

```text
40,41,42,43,44
```

Keep this seed set in one place so CLI help, tests, and demo output do not drift.

Proposed constant:

```python
DEMO_SEEDS = [40, 41, 42, 43, 44]
DEMO_CONFIG_A = "stub:naive"
DEMO_CONFIG_B = "stub:procedure"
```

Location: `solvent/cli/main.py` or a new `solvent/demo.py`.

## 6. Trace Schema Changes

The current trace is enough for scoring, but not quite enough for the viewer's event inspection goal. In particular, `submitted` currently records only `{"job_id": ...}`. The viewer cannot show the submitted artifact.

### 6.1 Add Safe Artifact Preview

Update `Environment.submit` in `solvent/env/env.py`:

Current:

```python
self._emit("submitted", {"job_id": job_id}, self.config.tool_call_cost)
```

Proposed:

```python
self._emit(
    "submitted",
    {
        "job_id": job_id,
        "artifact_preview": _artifact_preview(artifact),
        "artifact_size": len(artifact),
        "artifact_sha256": _artifact_sha256(artifact),
    },
    self.config.tool_call_cost,
)
```

Rules:

- Artifact preview is public agent output, so it is safe to trace.
- Cap preview length, e.g. 4 KiB, to avoid huge JSONL lines later.
- Include `artifact_truncated: true` when capped.
- Keep full artifact out of the trace for future larger tasks.
- Do not include hidden expected outputs, rubrics, costs, or reservation prices.

For v0.3 `data_clean`, 4 KiB is enough to show the full cleaned CSV in practice.

### 6.2 Keep Verifier Details As Is

`verified_pass` / `verified_fail` already include:

- `job_id`
- `score`
- check names
- check pass/fail state
- check messages

The viewer should consume these directly instead of creating a second verifier summary.

### 6.3 Do Not Break Scoring

`solvent.scoring.events.facts_from_events` should ignore the new `submitted` payload keys. No score formula should change in v0.3.

## 7. Run Manifest Design

The current `compare` command writes only aggregate `summary.json`. The viewer needs to know which traces belong to which config/seed/red-team condition.

Add `manifest.json`.

The manifest must include all runs, not only the red-team-off runs used for the headline comparison. The default demo has 20 run artifacts:

```text
2 configs * 5 seeds * 2 red-team states = 20 traces
```

Proposed shape:

```json
{
  "schema_version": "solvent_demo_v0_3",
  "created_by": "solvent <version>",
  "summary_path": "summary.json",
  "viewer_entry": "viewer/index.html",
  "configs": ["stub:naive", "stub:procedure"],
  "seeds": [40, 41, 42, 43, 44],
  "redteam_paired": true,
  "runs": [
    {
      "config_id": "stub:naive",
      "seed": 40,
      "redteam_enabled": false,
      "trace_path": "seed-40-stub-naive-redteam-off.jsonl",
      "scorecard_path": "seed-40-stub-naive-redteam-off.scorecard.json",
      "view_path": "viewer/data/seed-40-stub-naive-redteam-off.view.json"
    }
  ]
}
```

Implementation notes:

- Store paths relative to the demo directory.
- Keep `summary.json` as the aggregate comparison artifact for compatibility.
- Generate one canonical view JSON per trace for tests, diffs, and the future API seam.
- Also generate `viewer/data.js`, embedding `manifest`, `summary`, and all trace view models in `window.SOLVENT_DATA`.
- In v0.3, embedding all 20 tiny traces is acceptable; the payload is expected to be tens of kilobytes. Lazy loading can return later when traces become large.
- Single-source `created_by` from package metadata or `solvent.__version__`; do not hardcode a version string in multiple places.

```json
      "id": "dc-42-0",
      "type": "data_clean",
      "brief": "...",
      "arrival_tick": 0,
      "inputs": {}
    }
  }

```

### 8.3 Event Presentation Rules

Map raw event kinds to readable titles while keeping raw details visible.

Examples:

- `board_seen` -> "Board seen"
- `inspected` -> "Job inspected"
- `bid_made` -> "Bid made"
- `bid_accepted` -> "Bid accepted"
- `bid_declined` -> "Bid declined"
- `manipulation_attempt` -> "Manipulation attempt"
- `manipulation_conceded` -> "Discount conceded"
- `manipulation_resisted` -> "Discount resisted"
- `submitted` -> "Artifact submitted"
- `verified_pass` -> "Verifier passed"
- `verified_fail` -> "Verifier failed"
- `paid` -> "Payment credited"
- `invalid_action` -> "Invalid action"
- `overhead_charged` -> "Overhead charged"
- `terminated` -> "Episode terminated"

### 8.4 Job Extraction

Build a public job lookup from:

- `board_seen.payload.jobs`
- `inspected.payload.job`

Never regenerate hidden market data for the viewer. The viewer should show only what was public in the trace plus scorecard metrics.

### 8.5 Verification Pairing

For each `submitted` event, link the next verifier event for the same `job_id` if present. Also keep verifier events as timeline events in their own right.

This makes the event detail panel useful when clicking either:

- `submitted`: show artifact preview and eventual check result
- `verified_pass` / `verified_fail`: show checks and score

## 9. Viewer Builder

Create `solvent/viewer/build.py`.

### 9.1 Function API

```python
def build_viewer(
    trace_dir: Path,
    summary: dict[str, Any],
    runs: list[RunArtifact],
) -> Path:
    ...
```

Return the path to `viewer/index.html`.

### 9.2 Responsibilities

1. Create `trace_dir / "viewer"`.
2. Copy static files from `solvent/viewer/static/`.
3. Create `viewer/data/`.
4. For every trace, write one `.view.json`.
5. Write `manifest.json` at the trace directory root.
6. Write `viewer/data.js` containing `window.SOLVENT_DATA = ...`.
7. Optionally write `viewer/data/manifest.json` and `viewer/data/summary.json` as convenience copies, but the browser should not depend on `fetch()`.
8. Ensure all generated JSON is deterministic:
   - sorted keys
   - no wall-clock timestamps
   - relative paths

### 9.3 Determinism Rule

v0.2 traces are byte-deterministic for the same seed/config/path. v0.3 generated JSON should also be deterministic per demo directory. Do not add generated timestamps to manifest files.

Relativize every path embedded in view artifacts, including `Scorecard.trace_path`, against the demo directory before serialization. This keeps the viewer portable across machines and prevents absolute user paths from leaking into `data.js`.

## 10. Static Viewer UI

Use plain HTML, CSS, and JavaScript. No React build step in v0.3.

### 10.1 Layout

Desktop:

```text
+--------------------------------------------------------------+
| Top bar: Solvent v0.3 Demo | config/seed/red-team selectors  |
+---------------+--------------------------+-------------------+
| Event timeline| Event detail             | Scorecard/curve    |
|               | Artifact/checks/payload  | Comparison metrics |
+---------------+--------------------------+-------------------+
```

Mobile:

- Top selectors remain sticky.
- Use tabs: `Trace`, `Detail`, `Score`.
- Avoid dense side-by-side columns below roughly 800px width.

### 10.2 Visual Design

This is an operational eval tool, not a marketing page.

Use:

- restrained colors
- dense but readable tables
- 8px-or-less card radii
- clear event badges
- small inline SVG or CSS-only chart for the balance curve
- no oversized hero section
- no decorative gradients or illustration

### 10.3 Required Viewer Interactions

1. Select config.
2. Select seed.
3. Toggle red-team off/on.
4. Click event.
5. Jump to manipulation event.
6. Jump to failed verifier event if present.
7. Switch between `Trace` and `Compare`.

### 10.4 Comparison Tab

Read `window.SOLVENT_DATA.summary` and render:

- table by metric
- config A mean/std/n
- config B mean/std/n
- paired delta mean/std/n

Highlight:

- `Net revenue (baseline, red-team off)` delta
- `Manipulation-resistance loss (red-team on minus off)` delta
- pricing regret delta

Use the existing summary shape from `solvent.scoring.compare.ConfigSummary`; do not invent new aggregate math in JavaScript. The row labels should make clear that headline net revenue is computed from red-team-off runs, while manipulation resistance is the paired red-team-on/off stressor.

### 10.5 Balance Curve

Use the `balance_curve` in the view JSON. The JS should only render points; Python should derive the values.

For v0.3, a simple SVG polyline is enough:

- x-axis: event index or tick
- y-axis: balance
- tooltip or selected-event marker

## 15. Risks and Mitigations

| Risk | Mitigation |
|---|---|
| Viewer starts duplicating score logic | Keep all aggregate math in Python; JS only renders serialized scorecards and summaries |
| Trace leaks hidden fields through view JSON | Build view models only from trace public events and scorecard metrics; add explicit hidden-field tests |
| Static browser cannot fetch local JSON under strict `file://` behavior | Generate `viewer/data.js` and load it with a normal script tag; keep `solvent view --serve` only as a convenience |
| `main.py` becomes too large | Move compare/demo orchestration to `solvent/demo.py` and viewer writing to `solvent/viewer/build.py` |
| Artifact previews get too large later | Cap preview length now and include hash/size/truncation metadata |

## 16. Definition of Done

v0.3 is complete when a reviewer can run:

```bash
python3 -m solvent.cli.main demo --trace-dir runs/demo-v0_3
python3 -m solvent.cli.main view runs/demo-v0_3
```

and immediately see:

1. `stub:naive` vs. `stub:procedure`
2. mean +/- std over five paired seeds
3. a dollar delta
4. red-team-on/off manipulation-resistance loss
5. a clickable event trace
6. artifact and verifier details for submitted work
7. the exact scorecard explaining whether the loss came from selection, pricing, delivery, support/manipulation, or coherence

That is the v0.3 demo: not more world complexity, but the existing attribution claim made visible.
