# Solvent v0.5 Technical Implementation Plan — Long-Horizon, Multi-Model, Cost-Optimized Experiments

**Scope:** turn the v0.4 attribution backend into a **repeatable experiment platform** — run many models over a long business horizon, cheaply and reproducibly, then emit a Vending-Bench-style findings report + demo.
**Current baseline:** v0.4 ships the real LLM harness (Anthropic only), the delivery-tool economy, simulated delivery from a frozen menu, two-economy scoring, provenance/replay, and per-stage attribution — but only at *demo scale* (horizon 5, one provider, crude turn-capped API context, quadratic sidecar history, no cost guards).
**Target commands:** `solvent estimate <experiment.yaml>` (dry-run cost), `solvent experiment run <experiment.yaml>` (resumable matrix run), `solvent findings <experiment-dir>` (leaderboard + report + multi-model viewer).
**Primary output:** a config-driven experiment that produces, per model, a distribution of net revenue + per-capability attribution + compute-efficiency over a long horizon — and a written findings doc + interactive leaderboard, the way VB2 did.
**Out of scope (defer to v0.6 "validate"):** the SWE-Lancer anchor (C.7); concurrency (`max_wip`) + live direct-delivery worker mode; RL training loop (Phase 3); reputation dynamics beyond the static arrival/expiry stream (optional within v0.5b).

> **Roadmap note.** This milestone is **distinct from the PRD's original C.17 "v0.5 = SWE-Lancer anchor."** Per the reconciliation in C.17 / C.20.8, the experiment-&-findings platform (this doc) becomes **v0.5** and the SWE-Lancer anchor + concurrency/direct-delivery move to **v0.6 (validate)** — you run the experiments first, then validate against the anchor. This doc *activates and scales* the v0.4b time model (C.20.5); it does not re-introduce it.

> **The goal in one line:** *make it possible to ask "which models can profitably run this freelance business over a month, and which capability does each one fail on?" — run it for $X you chose in advance, and get back a VB-style leaderboard plus the attribution VB can't produce.*

---

## 1. v0.5 Objective

v0.4 proved a real model can run the business and be attributed per-capability. v0.5 makes that **usable as a research instrument at scale**. Four capability pillars, each closing a concrete gap that blocks the experiment you want:

1. **Long horizon (time, not turns).** Switch the default from the turn-coupled 5-tick demo to **business-time** measured in sim-days, advanced by *work duration* (C.20.5), with a long job stream. This is the VB2 long-horizon coherence analog.
2. **Cost-optimized.** Make long runs *affordable and safe*: token-bounded context (replacing today's crude last-20 prompt cap), linear sidecars/in-memory history, prompt caching, simulated delivery (already in v0.4), and **hard cost budgets with pre-flight estimation** so nobody gets a surprise bill.
3. **Multi-model.** A provider-neutral model-client layer so Claude, GPT, Gemini, **and the cheaper open models (Kimi K2.6, latest GLM, MiniMax)** all run through the same boundary with normalized token accounting and correct pricing. Most of the cheaper models expose **OpenAI-compatible** endpoints, so they share one adapter — and being cheap, they're the recommended **starting point** for shaking out the pipeline before spending on frontier models.
4. **Findings & demo.** A repeatable experiment runner (matrix, resumable, parallel, cost-tracked) and a findings generator: a VB-style leaderboard + the per-capability decomposition + balance-over-time + money-shot traces, as a report and an interactive viewer.

Everything stays behind the v0.4 env/harness/scorer boundary — v0.5 is additive (a time model, a context manager, more clients, an orchestrator, a report generator), not a rewrite of scoring.

## 2. Source-of-Truth Reconciliation

PRD as product truth — **C.20.5** (business time), **C.13** (memory ablation, thin harness), **C.4A** (reactive market / arrival / expiry), **C.9 / C.20.2** (burn & the two economies), **C.10 / C.11** (metrics & references), **C.12** (characterization, dev/test split), **C.15** (the demo), **A.7** (attribution-not-realism positioning) — and the current code as implementation truth.

The boundary is unchanged: the harness drives the env through the `ToolAdapter` (v0.4 §6.0), the env writes a JSONL trace + LLM sidecar, the scorer replays it. v0.5 adds *new harness internals* (time-aware loop, context manager, more clients), *new env behavior* (calendar-driven arrivals/expiry, real business-time clock), and *new tooling on top* (estimate / experiment / findings).

> **Correction to a tempting claim: the scorer is NOT formula-stable here.** The per-stage *signal* math (selection/pricing regret shapes) is stable, but their **inputs change** once the market is a reactive arrival/expiry stream: reachability is no longer "arrived before `horizon_ticks`," and the reference **optimum is no longer a static knapsack** (PRD C.11.3 / C.4A.5 — it becomes the optimal policy of the seeded decision process: exact DP for small instances, knapsack as an *upper-bound relaxation* for large). So v0.5b carries a real **dynamic-scoring workstream** (§7.4). Skipping it makes long-horizon results look precise but **misattributed** — the opposite of the point.

## 4. Acceptance Criteria

v0.5 is done when all hold.

1. **Real business-time clock.** `Clock` is rewritten to an **integer-unit `business_time`** (minutes; days are display); an episode runs to `horizon` with jobs on a seeded `arrival`/`expiry` calendar; `deliver` advances time by tool `duration` (once, not a tick loop); `advance_to_next_event` jumps to the next event; overhead is **prorated by elapsed business time**. **`end_tick` stays valid** (deprecated alias in business-time mode; unchanged in tick mode) so all v0.1–v0.4 tests pass. (A clock rewrite, not a flag flip — §7.1.)
2. **Dynamic scoring, reference-relative (no misattribution).** `fraction_of_optimal` is computed against the **reactive realizable reference** (C.11.2–3). An expired-unbid job is charged as selection/attention regret **iff the reference policy would have won it under the same constraints**: a winnable job the agent **idled out counts as regret**; an **exogenously-unreachable** expired job does **not** (§7.4).
3. **Bounded context — three concerns.** The **API payload** stays under `ctx_window_tokens` across a long run (token-based, not the crude last-20); the **sidecar grows linearly** (not quadratically) in turns; `context_policy` is **identical across providers** and **distinct from the `memory` ablation** (§8).
4. **Prompt caching on.** Stable prefix (system + tool schemas + history prefix) is cached where the provider supports it; verified by a non-zero cache-read counter.
5. **Multi-model.** `claude-*`, `gpt-*`, `gemini-*`, and `kimi-*`/`glm-*`/`minimax-*` all run through the same `ToolAdapter` (OpenAI-compatible ones share one adapter), with normalized `TokenUsage(input, output, cache_read, cache_write)`.
6. **Versioned pricing registry — schema, not hardcoded rates.** Every model is a registry entry with required provenance (`source_url`, `verified_date`, `version`); the one known correction is Opus 4.8 = $5/$25; **unknown models fail loudly**, never $0. Acceptance checks the schema + fail-loud, not specific numbers.
7. **Pre-flight cost estimate.** `solvent estimate <experiment.yaml>` prints projected cost (per model, per cell, total) from a turn/context model **before** spending, and the experiment refuses to start above its `budget_usd` without `--yes`.
8. **Hard budget guard with reservation.** A running experiment **aborts cleanly** (preserving completed cells) at `budget_usd`, and **spend reservation** prevents `parallelism` in-flight cells from collectively overshooting the cap.
9. **Resumable matrix.** `solvent experiment run` expands models × seeds × samples × conditions × ablations with a `pending/running/completed/failed/skipped_budget` state machine, and **resumes** without re-running completed cells or re-spending.
10. **Re-scoring is free.** Re-scoring / re-aggregating any experiment makes **zero** model API calls (replay from sidecars).
11. **Findings output.** `solvent findings <dir>` emits a **VB-style leaderboard** (per model: net revenue mean ± std and **min**, fraction-of-optimal, jobs delivered, days-until-insolvent, % of horizon), the **per-capability decomposition** across models, a **compute-efficiency** column, a `findings.md` report, and a multi-model viewer (leaderboard + per-model balance curves + money-shot traces).
12. **Reproducibility at scale.** Every cell records full provenance (model, pricing version, menu checksum, context_policy/caching, horizon, market params, seed, sample index); same config + seeds → same matrix.
13. **All v0.1–v0.4 tests pass**; the v0.4 stub demo and viewer still work.

## 7. Time Model & Long-Horizon Market (v0.5b)

### 7.1 Deliverable (i) — Clock/Calendar migration (PRD C.20.5)

> **This is a real migration, not a flag flip.** v0.4b laid a *seam* (`work_time_enabled`, `_advance_business_time`) but the *implemented* clock is still an **integer tick counter** ([clock.py](solvent/env/clock.py)) and `_advance_business_time` just loops `clock.advance()` one tick per duration unit. v0.5 replaces it with a real business-time clock.

Concrete deltas (rewrite [clock.py](solvent/env/clock.py), extend [models.py](solvent/env/models.py) / [env.py](solvent/env/env.py)):

- **Integer time units, not float days (Codex P2).** `business_time` is an **integer count of a base unit** (minutes — mirroring VB's 5min/25min/75min/5h steps), *not* a float of days. `horizon`, `arrival`, `expiry`, and tool `duration` are all in that integer unit; "days" is a **display** convenience (`minutes / 1440`). This avoids float-boundary nondeterminism at arrival/expiry/horizon comparisons — consistent with the codebase's Decimal/seeded determinism ethos. Terminate at `business_time ≥ horizon` **or** insolvency.
- **Advance by work:** each `deliver` advances `business_time` by the tool's `duration` (integer units), **once** — not a per-tick loop. Deliberation is free in time (model turns never move the clock; explicitly *not* VB's tool-call-advances-time).
- **`advance_to_next_event` tool:** jumps `business_time` to the **next calendar event** (next `arrival` or `expiry`), so sparse-job horizons don't burn a turn per empty unit.
- **`end_tick` compatibility (Codex P2).** Two clock modes: **tick mode** (v0.4 default, `work_time_enabled=False`) is **unchanged** — `end_tick` advances one tick — so *all v0.1–v0.4 tests, stubs, and traces keep passing*. **Business-time mode** (experiments) adds `advance_to_next_event` and keeps `end_tick` as a **deprecated alias** for it (advance-to-next-event), so existing harnesses/traces still work; the new tool is preferred and documented.
- **Overhead is prorated by elapsed business time** (`overhead_per_unit × Δunits`), charged on each advance — not a flat per-tick fee. The VB daily-fee analog, time-correct.

### 7.2 Long job stream (extends [market.py](solvent/env/market.py), C.4A)

Replace the fixed `market_size` board with a **seeded job stream over the horizon**:
- Arrivals on a seeded schedule (`arrival_rate_per_day`, Poisson or fixed cadence) with `arrival_minute` and a seeded `expiry_minute` (a job leaves the board if not bid within its window — the attention-allocation pressure of C.4A.2). Days remain a reporting/config convenience; the internal clock stores integer minutes.
- Each job carries hidden `internal_difficulty` (from the difficulty distribution) + `reservation_price` + optional manipulation, as today.
- **Reputation (optional within v0.5b):** if `reputation_enabled`, condition later arrivals on delivered quality (C.4A.3). Keep behind the flag so the baseline stream is the simpler object.
- The scorer reconstructs the stream from provenance exactly as it reconstructs the board today (seed + market params) — no new leak surface.

### 7.3 New VB-style coherence signals

Because runs are now long, add the VB Table-1 analogs to the scorecard: **days-until-insolvent** (or "survived horizon"), **jobs delivered** (units-sold analog), **% of horizon active** (stagnation), and keep the existing dropped-jobs / loops / off-task meltdown signals (C.10) — these are exactly the long-horizon coherence failures VB exists to surface.

### 7.4 Deliverable (iii) — Dynamic scoring (the misattribution risk — PRD C.11.2–3 / C.4A.5)

Once the market is a reactive arrival/expiry stream, the static scorer ([scorecard.py](solvent/scoring/scorecard.py) `reachable_jobs` keys off `horizon_ticks`; the optimum is a static knapsack) **no longer computes the right reference**. v0.5b adds a scorer workstream so attribution stays honest:

- **Business-time facts** in `TraceFacts`: replay `arrival`/`expiry`, realized availability windows, which jobs **expired unbid**, active vs idle business time.
- **Reactive optimum** (C.11.3) — the reference policy: for small instances compute the exact optimal policy of the seeded decision process by DP/search; for large instances use the **knapsack as a labelled upper-bound *relaxation*** (it ignores expiry/discovery, so it can only over-state) — never call the relaxation "the optimum."

**Expiry semantics — define reachability against the *reference policy*, not the agent's realized path (Codex P1, the load-bearing fix).** Otherwise a lazy agent could **idle good jobs out** and dodge all selection regret ("I never reached them"). So:

- An expired-unbid job is charged as **selection / attention regret iff the capability-conditioned reference policy, under the same seed and constraints (budget, time, concurrency/WIP), *would have bid and delivered it***. That is precisely the realizable / regret-to-go reference (C.11.2, C.4A.5) — the agent is measured against *what a good policy could have won from the realized state*, not against its own choices.
- A job that **even the reference policy could not have reached** (exogenously unreachable — too many simultaneous arrivals, a window too short for any policy, budget already exhausted) is **not** charged to the agent.
- This makes idling-to-avoid-regret impossible: the reference *would* have taken the winnable jobs, so the agent eats the regret for letting them expire — exactly the attention-allocation pressure (C.4A.2) we want to measure.

**Acceptance (`test_dynamic_scoring.py`):** (a) `fraction_of_optimal` is computed against the reactive realizable reference; (b) a winnable job the agent idled out → **counts as selection/attention regret**; (c) an exogenously-unreachable expired job → **not** charged.

## 8. Context Management — three distinct things, not one (v0.5a)

**Correct the premise first.** It's *not* true that the harness sends unbounded history to the API. There are three separate concerns, and only one is bounded today:

| Concern | Today | Fix |
|---|---|---|
| **Prompt context** (sent to the API) | **already crudely capped** at `history[-20:]` ([model_client.py:185](solvent/harness/model_client.py)) — so API cost is ~linear, *not* quadratic | replace the turn-count cap with a **token-based** policy (below) |
| **Sidecar bloat** (replay log) | [llm.py:103](solvent/harness/llm.py) writes the **full history into every row** → **quadratic disk/IO** | write only the per-turn delta (request prefix once + per-turn obs/call/result) |
| **In-memory history** | unbounded list | bound it to what the policy needs |

`solvent/harness/context.py` — a provider-neutral `ContextManager` that builds each turn's message list under a **token budget** (`context_policy`, default ~24K — VB used ~30K):

- **`none`** — full history (short demo runs only).
- **`sliding_window`** — keep the last `ctx_window_tokens` of history (token-based, not turn-based — fixes the crude last-20).
- **`scratchpad`** — a structured running summary (open jobs, priced/delivered, balance, outcomes) carried in place of raw old turns; most token-efficient.

**Naming (Codex P2): this is `context_policy`, NOT the `memory` ablation.** The harness already has `ABLATIONS = {memory, planner, procedure}` ([llm.py:14](solvent/harness/llm.py)) — `memory` there is a *scaffold the model controls* ("what it remembers"). `context_policy` is *infra deciding how much raw history to send*. They are orthogonal axes; keep them separate in the matrix or you confound "what the model remembers" with "how much history we feed it."

**Cross-model fairness invariant:** `context_policy` is part of the experiment config, applied **identically to every model** — never provider-specific — so a cost/capability gap is never an artifact of one model getting more context. With a token-bounded window, long-horizon **coherence** becomes a real test: the agent must use its scaffold deliberately, as VB's agents did.

**Acceptance tests (not just "a context manager exists"):** (1) the **API payload** stays under `ctx_window_tokens` across a long run; (2) **sidecar size** grows linearly (not quadratically) in turns.

## 9. Prompt Caching (v0.5a)

Structure every request **stable-prefix-first** (system → tool schemas → history prefix → volatile current observation) and cache the prefix:

- **Anthropic:** `cache_control: {type: "ephemeral"}` on the last stable block (reads ≈ 0.1× input). Verify `cache_read_input_tokens > 0`.
- **OpenAI / Google:** their automatic / explicit prompt-context caching, set in each client.
- The unified layer guarantees prefix stability (deterministic tool order, no per-request timestamps/IDs in the prefix) so caching actually hits.

**Caveat for cost comparison (ties to C.20.2):** caching availability/economics differ by provider, so a *compute-cost* comparison is partly a provider-pricing artifact. This is exactly why compute cost stays a **separate efficiency lens** (fraction-of-optimal per compute-$), never folded into the capability score. Report both; lead with capability.

## 10. Multi-Provider Model Layer (v0.5c)

### 10.1 Provider-neutral interface (`harness/providers/base.py`)

```python
class ModelClient(Protocol):
    def complete(self, request: ModelRequest) -> ModelResponse: ...

# ModelRequest: model, system_prompt, observation, tools (neutral schema),
#               messages (from ContextManager), temperature, cache_hint
# ModelResponse: tool_call, usage: TokenUsage(input, output, cached_read, cached_write)
```

`client_for_model(model_id)` dispatches by family to a small set of adapters, each a `(adapter, base_url, api_key_env, pricing)` registry entry:

| Family | Adapter | Notes |
|---|---|---|
| `claude-*` | `anthropic.py` (native) | best caching/thinking support |
| `gemini-*` | `google.py` (native) | native context caching |
| `gpt-*`, `kimi-*` / `moonshot-*`, `glm-*`, `minimax-*`, `fake`/`recorded` | **`openai_compat.py` (one adapter)** | OpenAI-compatible Chat Completions; **add a new cheap model = one registry row + pricing, not a new client** |

**The key simplification: Kimi K2.6 (Moonshot), the latest GLM (Zhipu), and MiniMax all expose OpenAI-compatible endpoints**, so a single `openai_compat.py` adapter parameterized by `base_url` + `api_key_env` + `model_id` serves all of them *and* OpenAI. Each adapter maps the neutral request to its tool-calling format via `schema_translate.py`, sets up that provider's caching, and **normalizes usage** (every provider reports tokens differently). Credentials per provider via env vars; `solvent doctor` checks each.

### 10.2 Tool-schema translation (`schema_translate.py`)

Our `TOOL_SCHEMAS` (v0.4, Anthropic-shaped) become the **neutral** definition; translate to OpenAI-style `tools`/`functions` (covers OpenAI + Kimi + GLM + MiniMax) and Google `function_declarations`. One source of truth for the action space; per-provider adapters only reshape it. (Default brain stays a latest Claude model per project guidance; the layer is provider-agnostic.)

### 10.3 Pricing, corrected and single-sourced (`pricing.py`)

- **Extend `TokenUsage` first** — it has only `input/output` today ([pricing.py:10](solvent/env/pricing.py)); budget accuracy needs `TokenUsage(input, output, cache_read, cache_write)`. This is a **prerequisite** for the budget guard (§11.2), not a parallel task.
- **Data-driven, versioned registry — not hardcoded prices (Codex P2).** Each model is a registry *entry* with **required provenance fields**: `{input, output, cache_read, cache_write, source_url, verified_date, version}`. The doc names no authoritative rate (I don't have verified Kimi/GLM/MiniMax/GPT/Gemini pricing); the *one* known correction is Opus 4.8 = $5/$25 (was wrongly $15/$75). Acceptance checks the **schema + provenance**, not specific numbers.
- **Unknown model → raise**, never silent `$0` (today's failure mode, [pricing.py:29](solvent/env/pricing.py)). `pricing_table_version` recorded in provenance; `brain_cost(model, usage)` accounts for cached vs uncached tokens.
- **Cost staging:** cheap open models cost a fraction of frontier per token, so the canonical experiment (§12) is a **template** that runs them **first** to validate the pipeline at low spend, then adds frontier models once the matrix and findings are proven.

## 11. Cost: Estimate, Meter, Cap (v0.5a + v0.5d)

### 11.1 The cost model (`experiment/estimate.py`)

Cost is driven by **turns × context-per-turn × price**, not by the business horizon directly. The estimator computes, per (model, condition):

```
turns        ≈ jobs_over_horizon × turns_per_job + idle_advances        # jobs from arrival_rate_per_day × horizon_minutes / 1440
ctx_per_turn ≈ system + tool_schemas + min(history, ctx_window_tokens)  # bounded by the context manager
input_tokens ≈ turns × ctx_per_turn × (cache_discount if caching)
output_tokens≈ turns × output_per_turn
cost         = input × $in + output × $out                              # per pricing.py
```

`solvent estimate <experiment.yaml>` prints a per-model and total projection **before** any spend. (Calibrate `turns_per_job` and `output_per_turn` from one real run — see §15.)

### 11.2 Budget guard with **spend reservation** (`experiment/runner.py` + `state.py`)

Depends on §10.3's `TokenUsage(…, cache_read, cache_write)` + fail-loud pricing.

- Hard `budget_usd` per experiment. Every cell's real spend (from metered `TokenUsage`) accrues to a durable ledger.
- **Reservation, not just post-hoc accounting (Codex P1).** Before launching a cell the runner **reserves** its *estimated* cost against the cap; a cell only starts if `spent + reserved + cell_estimate ≤ budget_usd`. This is what stops **N parallel in-flight cells from collectively overshooting** the cap (post-hoc checks can't — they only see completed cells). On completion, reconcile reserved → actual.
- **In-episode kill switch (Codex P2).** Reservation bounds *parallel* overshoot but not a single runaway episode (a model that loops/rambles past its estimate). So the **harness loop checks budget every turn**: after each turn's metered `TokenUsage`, if the cell's actual spend crosses `reserved + allowed_overrun` **or** global remaining budget ≤ 0, it **stops the episode and marks the cell `failed_budget`** (a terminal, resumable state). The partial trace is still scored for debugging and trace inspection, but it is **budget-censored** for leaderboard aggregation (§12.2 / §13.1).
- The runner stops launching new cells and **aborts cleanly** at the cap, preserving completed cells + resumable state.
- `--yes` required to start an experiment whose estimate exceeds `budget_usd`.

### 11.3 What does *not* help (be honest)

**The Batch API does not fit agentic episodes** — each turn depends on the previous, so an episode can't be a batch. The real throughput lever is **parallelism across independent episodes** (§12), not Batch. Don't budget for Batch's 50% discount on the agent loop; it doesn't apply.

## 12. Experiment Orchestration (v0.5d)

### 12.1 Declarative config (`configs/experiments/vb_style_v1.yaml`)

```yaml
name: vb_style_v1
# Start cheap to validate the pipeline at low spend, then add frontier models.
models: [kimi-k2-6:base, glm-latest:base, minimax-latest:base]   # cheap open models first
# then extend: [claude-opus-4-8:base, claude-sonnet-4-6:base, gpt-5.x:base, gemini-pro:base]
seeds: test                 # dev/test split; never tune on test
samples_per_seed: 3         # temp>0 variance (VB-style multiple runs)
conditions: [redteam_off, redteam_on]
horizon_minutes: 43200       # 30 sim-days; days are display/config convenience, minutes are internal truth
market: { task_mix: {data_clean: 0.5, extract: 0.5}, arrival_rate_per_day: 1.0, decoy_rate: 0.3, manipulation_rate: 0.2 }
context_policy: sliding_window   # infra: how much raw history we send — applied to ALL models (NOT the `memory` ablation)
ablations: [base]                # harness scaffolds: base | +memory | +planner | +procedure (orthogonal to context_policy)
caching: true
temperature: 1.0
budget_usd: 200             # hard cap
parallelism: 8              # concurrent episodes, rate-limit-aware
```

### 12.2 Matrix, resume, parallelism — explicit state machine (`matrix.py`, `runner.py`, `state.py`)

- Expand to cells = `models × seeds × samples × conditions × ablations`; each cell = one episode → trace + sidecar + scorecard.
- **Per-cell state machine** (durable, keyed by a deterministic cell id): `pending → running → {completed | failed | skipped_budget | failed_budget}`. On restart, `completed` cells are skipped (no re-run, no re-spend); `running`-but-orphaned cells are reclaimed; `failed` cells are retried up to a cap; `skipped_budget` (reservation blocked launch) and `failed_budget` (in-episode kill switch, §11.2) cells are resumable if the cap is raised. Essential for long, expensive runs.
- **Spend reservation across parallelism (§11.2):** a `pending` cell only transitions to `running` if `spent + reserved + estimate ≤ budget_usd`, so up to `parallelism` in-flight cells cannot collectively overshoot.
- **Rate-limit aware:** per-provider RPM/TPM backoff (429 handling). Cells are independent episodes (each episode's turns stay sequential — see §11.3 on why Batch doesn't fit).
- Writes the experiment directory: all traces/sidecars/scorecards + `manifest.json` (every cell's provenance + state) + `ledger.json` (reserved vs actual spend).
- **Aggregation semantics:** primary leaderboards include only `completed` cells. `failed_budget` cells are excluded from capability means and instead surfaced in separate reliability/cost columns (`budget_kill_rate`, `completed_cells`, `censored_cells`) and in the report narrative. Debug views may show their partial scorecards, but findings never mix budget-censored partial episodes into the main business-performance distribution.

### 12.3 Statistical design (PRD C.10)

VB reported **high variance** — so report distributions, not point estimates: **mean ± std and min** (worst-case reliability, VB's "Net worth (min)") over `seeds × samples`, paired-by-seed for manipulation-resistance. Use **test** seeds for reported numbers. Make `samples_per_seed` a knob; surface confidence intervals in aggregation (`scoring/aggregate.py`).

## 16. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| **Surprise bill on a long run** | `solvent estimate` pre-flight + hard `budget_usd` guard + per-cell ledger + `--yes` gate (§11) |
| **Context window overrun / quadratic cost** | bounded context is a **v0.5a prerequisite**, not an optimization (§8); estimator + smoke confirm it holds |
| **Cross-model unfairness** | identical context policy, identical tool schema, identical market per seed across all models; compute cost kept a separate lens (C.20.2) |
| **Provider drift** (pricing/model changes) | verify rates at build; `pricing_table_version`; fail-loud on unknown model; `doctor` credential checks |
| **Batch-API mirage** | acknowledged: Batch doesn't fit agentic loops; throughput via parallelism, cost via caching + bounded context (§11.3) |
| **High variance hides signal** (VB saw this) | distributions over `seeds × samples`, report **min** + CIs, paired comparisons; `samples_per_seed` knob (§12.3) |
| **Long run dies midway** | resumable per-cell state; re-scoring is free (replay) — no lost spend (§12.2) |
| **Scope creep** (reputation, SWE-Lancer) | reputation behind a flag inside v0.5b; SWE-Lancer anchor explicitly a parallel/after track |
| **Menu not yet characterized** | ship the v0.4 §8A calibrated-synthetic menu; characterization is the upgrade path, not a v0.5 blocker |
