# Solvent v0.4 Technical Implementation Plan

**Scope:** the **real LLM agent harness** plus the delivery-tooling business economy, on top of the v0.3 attribution backend
**Current baseline:** v0.3 ships deterministic stub harnesses (`stub:naive`, `stub:procedure`), the per-stage attribution scorecard, paired-seed comparison, the static trace viewer, and a flat per-tool-call burn proxy
**Target commands:** `solvent run --agent claude-opus-4-8:+procedure --seed 42`, `solvent characterize --validate-menu` (v0.4a) / `--generate-menu --seeds dev` (v0.4-full), `solvent compare --a claude-opus-4-8:base --b claude-opus-4-8:+procedure --seeds test --viewer`
**Primary output:** a model-agnostic ReAct harness running against the *unchanged* environment boundary, where **delivery compute is a real in-sim business cost** and the **brain's own compute is metered but reported separately**; a characterization-backed delivery menu; a dev/test seed split; and a second task type
**Out of scope (→ v0.5):** concurrency / multi-task (`max_wip > 1`); live "direct delivery" worker mode; the SWE-Lancer anchor; RL training loop

> **This is the milestone where Solvent stops being driven by scripted stubs and is driven by a real model.** Everything else in v0.4 (the delivery economy, characterization, dev/test split, second task type) exists to make that real harness *measurable and fair*. The defining deliverable is the LLM agent harness.

---

## 1. v0.4 Objective

v0.4 turns Solvent from "a proven attribution machine exercised by deterministic stubs" into "a proven attribution machine exercised by a **real, swappable LLM harness** under a realistic cost model."

The v0.3 system can already:

1. Run a seeded static `data_clean` freelance episode and emit a JSONL trace.
2. Decompose the economic outcome into selection / pricing / delivery / support / coherence signals against hidden ground truth.
3. Compare two configs over paired seeds with a dollar delta and manipulation-resistance loss.
4. Render a clickable viewer.

But every config in v0.3 is a hardcoded `StubHarness` mode, and "cost" is a flat `tool_call_cost` placeholder because stubs generate no tokens.

v0.4 adds, in priority order:

1. **The real LLM agent harness (the headline).** A thin, model-agnostic ReAct loop that drives any chat/agent model through the *same* env tool API the stubs use, with token metering and the memory / planner / procedure ablation knobs (PRD C.13). Config IDs become `<model>:<ablation>` (e.g. `claude-opus-4-8:+procedure`).
2. **The brain / delivery-tooling split and two economies** (PRD C.20.2–C.20.3). The brain's tokens are metered but **not** debited from the in-sim balance; the **delivery tool** the agent chooses to do the work has a price that **is** debited as a business input. `est_cost` becomes the *realized, debited* delivery price.
3. **The frozen, calibrated-synthetic delivery menu** (PRD C.20.4–C.20.4.2). A hidden ground-truth table mapping `(tool_model, difficulty) → (pass_rate, duration)`, plus a public per-tool `price`; the agent sees only coarse capability/speed proxies and must estimate reliability.
4. **The characterization run** (`solvent characterize`, PRD C.12) producing per-stage difficulty profiles and the per-model delivery profile that backs the menu and the realizable reference (PRD C.11.2).
5. **A dev/test seed split** (PRD C.17) — never tune on test.
6. **A second task type, `extract`** (PRD C.6) — to confirm the machine generalizes beyond `data_clean`.
7. **(v0.4b) real business time and reputation/reactivity** (PRD C.20.5, C.4A) — staged after the harness and economy are stable.

The goal is *not* to add world complexity for its own sake. It is to make the real model the system under test, charge the costs a real AI-using freelancer would pay, and keep every per-stage attribution claim from v0.3 intact and now meaningful for a real agent.

## 2. Source-of-Truth Reconciliation

Use `phase1_prd_tdd.md` as the product source of truth (especially **C.13** harness, **C.20** economic model, **C.12** characterization, **C.11.2** realizable reference, **C.6** task types, **C.4A** reactivity) and the current codebase as the implementation source of truth.

The PRD already specifies the harness design and the v0.4 economic upgrade in detail; v0.4 is mostly *building what C.13 + C.20 describe*, not inventing new product.

### 2.1 What is reused vs. what is genuinely new

The **env's deterministic *resolution* logic is unchanged** — `bid` still clears iff `bid ≤ reservation_price`, money is still moved only by deterministic checks, and the scorer still replays the JSONL trace. That is the load-bearing reuse, and it is why this is not a rewrite. **But "drop-in" overstates it:** today there is *no* public tool boundary — `StubHarness.run(env)` simply calls Python methods (`env.list_jobs()` …) directly, and the env communicates failure by **raising** (`InvalidActionError`, `AlreadyTerminatedError`, `UnknownJobError`). A model cannot consume raised exceptions or call Python methods. So three things must be **built new**, not reused (promoted to first-class deliverables — Step 1, §6.0):

1. **A public tool adapter** — JSON tool schemas + an action dispatcher (model tool-call → env method).
2. **An observation serializer** — env state → the text/JSON the model sees (public fields only).
3. **An error envelope** — raised exceptions → structured `invalid_action` observations the model can react to (this path feeds the coherence axis, so it is load-bearing, not cosmetic).

What also changes: new tools (`list_models`, `deliver`), a new business debit (delivery-tool price), a **normalized delivery fact** so `deliver` outcomes feed the existing scorer (§7.5), and a **trace provenance contract** so menus/difficulty/pricing are replayable (§8B). The per-stage *formulas* are untouched; their *inputs* gain a normalization layer.

Concretely, today: `StubHarness.run(env)` drives direct methods; `harness_from_config_id()` in `solvent/demo.py` maps a config id to a harness. A real harness is a new class with the same `.run(env)` contract — but it drives the env *through the new adapter*, not by poking methods.

### 2.2 What stays frozen from v0.1–v0.3

Do not regress these:

- **Determinism:** `same seed + same policy → identical trajectory` (PRD C.4A). The env stays seeded; only the *policy* (the model) becomes stochastic — see 3.1.
- **Hidden/public boundary:** `reservation_price`, `est_cost`, `internal_difficulty`, `rubric`, `is_decoy`, and the per-model outcome table are never returned by a tool (PRD C.20.4.2).
- **Deterministic money / no LLM judges payment:** a sale clears iff `bid ≤ reservation_price`; a delivery's outcome is a seeded draw or a deterministic verifier — never an LLM's opinion (PRD G3).
- The v0.3 stub world remains runnable and the v0.3 viewer/compare keep working.

## 3. v0.4 Acceptance Criteria

v0.4a / v0.4-full are done when all of the following hold.

1. **A real model can run an episode end-to-end via the harness.** `solvent run --agent claude-opus-4-8:base --seed 42` drives a live model through the env tool API, meters its tokens, writes a JSONL trace, and produces a scorecard — with no change to the env's deterministic resolution code.
2. **Config-id dispatch** distinguishes `stub:*` (v0.3 stubs) from `<model>:<ablation>` (the LLM harness). Both run through the same `run_episode` / `run_compare_artifact` path.
3. **Ablation knobs work:** `base`, `+memory`, `+planner`, `+procedure` each produce a measurably different policy on the same seed (PRD C.13).
4. **Two economies are separated and reported** (PRD C.20.2): the in-sim **business** balance debits only overhead + delivery-tool price; the **brain's** real token cost is tracked and reported as a separate efficiency figure, never debited from the business balance.
5. **The agent chooses a delivery tool.** `list_models()` returns coarse public specs; `deliver(job_id, model)` resolves the outcome from the hidden frozen menu, debits the tool price, and credits revenue on pass. The agent never sees `pass_rate` or `internal_difficulty`; there is no hidden per-difficulty cost in v0.4a.
6. **The frozen delivery menu exists as data, behind the characterization schema** (PRD C.20.4.1): swapping a hardcoded calibrated-synthetic menu for a `characterize`-produced one is a data swap, not a code change.
7. **Characterization gates, split by stage** (resolves the required-vs-optional tension):
   - **v0.4a gate:** v0.4a ships the **hand-authored calibrated-synthetic menu** (§8A.7), and `solvent characterize --validate-menu` checks an existing menu against the four constraints + floor/ceiling on the dev split. Full corpus→menu *generation* is **not** a v0.4a blocker.
   - **v0.4-full gate:** `solvent characterize --generate-menu --seeds dev` runs the §8A.4 procedure end-to-end, emitting a per-`(tool, difficulty)` delivery profile **and** a per-stage brain profile in the menu schema, with floor/ceiling pruning (PRD C.12).
8. **Dev/test seed split** exists; reported comparisons use the test split; characterization/tuning uses dev. The split is recorded in config and in the trace.
9. **Second task type `extract`** has a generator + deterministic verifier and is scored by the existing per-stage machinery.
10. **Reproducibility under a stochastic policy** (PRD C.4A.4): at temperature 0 a run is replayable from recorded model responses; at temperature > 0 the harness supports N samples/seed and the scorer reports distributions. Re-scoring a saved trace makes **no** API calls.
11. **Hidden fields still never leak** into traces or viewer artifacts (extend the v0.3 guardrail to `internal_difficulty` and the per-model outcome table).
12. **All v0.1–v0.3 tests still pass**, and the v0.3 stub demo + viewer still work unchanged.

### 3.1 v0.4b Acceptance Criteria

v0.4b is a follow-on gate after v0.4a / v0.4-full: business time advances by work duration with job arrival/expiry; reputation conditions the future board (PRD C.20.5, C.4A.3).

## 6. The LLM Agent Harness (the headline)

`solvent/harness/llm.py` — deliberately thin and model-agnostic (PRD C.13).

### 6.0 The public tool adapter (NEW env boundary — Step 1 deliverable)

This does **not** exist today and is the largest genuinely-new engineering surface. It lives at the env boundary (`solvent/env/tool_api.py`) and is shared by every harness:

- **Tool schema** — a JSON schema per tool (name, args, types) the model is given for tool-calling. Source of truth for the action space (§15).
- **Observation serializer** — `observe(env) -> dict/text` exposing **public fields only** (board, balance, in-progress, model menu, time/horizon). Reuses `Job.to_public()`; never serializes hidden truth (§7.4, §8B).
- **Action dispatcher** — `dispatch(env, tool_call) -> observation`: validates the call against the schema, invokes the env method, and **converts raised exceptions into structured `invalid_action` observations** (the error envelope). The env keeps raising internally; the adapter is the single place exceptions become observations.
- **One-tool-per-turn** contract; malformed/invalid calls become `invalid_action` (existing event path) → cost a turn → surface in coherence scoring, exactly as for stubs.
- **Invalid-action ownership:** env-originated invalids emit exactly once from the env before raising; adapter-originated failures (schema validation, malformed tool call, unknown tool name) emit exactly once from the adapter. The adapter must not duplicate an `invalid_action` already emitted by the env.

The stub harness is refactored to drive the env *through this adapter too*, so both harnesses share one boundary and one error envelope (keeps the coherence axis identical across stub and LLM).

### 6.1 The loop

ReAct, driven through the §6.0 adapter:

```text
api = ToolAdapter(env)                  # the §6.0 public boundary; hidden truth never crosses it
while not env.terminated():
    obs   = api.observe()               # public board, balance, in-progress, model menu, time/horizon
    step  = model.act(obs, history)     # ONE tool call (+ optional reasoning); tokens metered
    obs2  = api.dispatch(step)          # validate → env method → structured observation (or invalid_action)
    record(step, obs2)                  # append to ReAct history + trace
    # end_tick when the model yields
```

- **Exactly the adapter's tool set (C.5 + v0.4 additions).** No privileged access: `list_jobs, inspect_job, clarify, bid, list_models, deliver, respond, check_balance, list_in_progress, end_tick`.
- **One tool call per turn**, parsed from the model's tool-call output. Malformed/invalid calls become `invalid_action` observations via the error envelope (§6.0) — never an uncaught exception that crashes the loop.

### 6.2 Ablation knobs (the experiment dimension — PRD C.13)

Config id = `<model>:<ablation>`, e.g. `claude-opus-4-8:+procedure`. Ablations compose:

- `memory`: `none` | `scratchpad` (CRM-style running summary of in-progress jobs and past outcomes).
- `planner`: `none` | `propose-then-act` (a select/price/tool plan before acting).
- `procedure`: `none` | `forced double-check` — research the brief + estimate reservation and delivery-tool fit *before* quoting/delivering (Project Vend 2's highest-leverage change; the headline ablation).
- `model`: any chat/agent model (the brain). Default brain = a latest, capable Claude model; harness stays provider-agnostic.

`base` = all knobs off. Knobs are prompt/scaffold changes only; they never touch the env.

**Config-id grammar (migration from v0.3).** Today the CLI takes `--agent stub` plus a separate `--stub-mode` (`solvent/cli/main.py`). v0.4 unifies these into a single `--agent <family>:<spec>` grammar:

- `<family>` = `stub` | a model id (e.g. `claude-opus-4-8`).
- `<spec>` for stubs = the mode (`stub:naive`, `stub:procedure`) — preserves v0.3 behavior.
- `<spec>` for models = `base`, or `+`-joined ablations: `claude-opus-4-8:+memory+procedure` (order-insensitive; deduped). `base` and any `+knob` are mutually exclusive.
- **Backward compatibility:** `--agent stub --stub-mode naive` keeps working and is normalized to `stub:naive`; `harness_from_config_id` dispatches on `<family>` (`stub:*` → `StubHarness`, else → `LLMHarness`). The composed config id is what is recorded in the trace and scoreboard.

### 6.3 Metering, pricing, and the two economies (PRD C.20.2–C.20.3)

- The harness reads token usage from each model response and accumulates **brain compute cost = tokens × price** from `env/pricing.py`. This is the **compute economy** — tracked, reported as an efficiency figure, **never debited from the in-sim balance.**
- The only things debited from the **business** balance are `overhead_per_tick` and the **delivery-tool price** returned by `deliver` (6 / 7). Revenue is credited on a passing delivery.
- The flat v0.3 `tool_call_cost` drops to `0.0` for LLM configs (it was a stub-era stand-in for token burn; with real metering it would double-count — PRD C.9 revised note).

### 6.4 Determinism, temperature, and record–replay (PRD C.4A.4)

The env is deterministic given the seed; the policy is not. To preserve reproducibility and re-scorability:

- The harness **records each model request/response (with token usage)** into a sidecar (e.g. `*.llm.jsonl`) keyed to the trace.
- **Replay/re-score makes no API calls** — `score_trace` and `solvent replay` read the recorded outputs. (Acceptance #10.)
- **Temperature 0** → a run is effectively reproducible from records. **Temperature > 0** → support `--samples N` per seed; the scorer reports mean ± std (PRD C.4A.4, the "average across N runs" property).
- Provider/model nondeterminism that survives temp 0 is acknowledged and absorbed by the sampling/reporting, never by weakening the env.

### 6.5 Model client

`solvent/harness/model_client.py` — a minimal provider-agnostic interface (`act(messages, tools) -> {tool_call, usage}`). Default implementation targets the Claude API; a `RecordedClient` / `FakeClient` backs tests and replay. Keep it free of scoring logic. (See the `claude-api` skill for current model IDs and pricing when wiring `env/pricing.py`.)

## 7. The Delivery Economy

### 7.1 Brain vs. delivery tooling (PRD C.20.3)

Model the agent as an **AI-using freelancer/firm**: the brain decides *which job to take, what to bid, and which delivery tool to use*; the chosen tool does the work. Tool compute is a business cost; brain compute is not. This makes `est_cost` the realized debited delivery price and resolves the v0.3 asymmetry (where `est_cost` was in the optimum but not the ledger).

### 7.2 New tools

| Tool | Signature | Semantics |
|---|---|---|
| `list_models` | `() -> [ModelPublic]` | the delivery menu the agent may hire: `name`, `price` (exact), `capability_proxy` (noisy — e.g. an advertised benchmark/tier), `speed_proxy`. **No** pass_rate, no per-difficulty cost. |
| `deliver` | `(job_id, model) -> DeliverResult` | env looks up the hidden `menu[model][internal_difficulty]`, **samples** pass/fail from the seeded RNG, **debits the tool price** from the business balance, (v0.4b) advances time by `duration`, and on pass **credits revenue**. Returns only the *realized* `{passed, price_charged, duration}` — never the probability. |

`deliver` replaces `submit` for the v0.4 **tool-mediated** mode. `submit(artifact)` is retained but only used by the future **direct-delivery** mode (v0.5 / anchor), gated by `delivery_mode: tool_mediated | direct` (default `tool_mediated`).

### 7.3 Simulated delivery is gaming-resistant by construction

In tool-mediated mode the agent does **not** produce an artifact — it picks a tool, and the outcome is a **seeded Bernoulli draw from the hidden menu**. The agent cannot fake a pass because it does not control the outcome at all; the menu + seed do. This is *more* gaming-resistant than artifact verification, and it is still fully deterministic given seed + policy. The deterministic-verifier guarantee (G3) lives where artifacts are actually produced: the **characterization run** that built the menu, and the future **direct-delivery/anchor** mode.

### 7.4 Information model (PRD C.20.4.2)

The per-job, per-model outcome table is **hidden ground truth**; the agent estimates it from public proxies. Carry the exact two-column boundary from C.20.4.2: public = `description/type/deadline` + `name/price/capability_proxy/speed_proxy`; hidden = `internal_difficulty/reservation_price` + per-`(model, job)` `true_pass_prob/true_duration`. Capability is a **noisy public proxy**, price is **exact and public** — concentrating difficulty on *will-it-succeed estimation*, not arithmetic.

### 7.5 Scoring compatibility layer (NOT just new event names)

The scorer today recognizes a submission **only** via `verified_pass`/`verified_fail` and a prior `submitted` (`solvent/scoring/events.py`), and derives delivery, gross score, support, and **coherence** (`dropped_jobs = accepted − submitted`) from those. `deliver` cannot just emit fresh `delivery_passed`/`delivery_failed` kinds or every one of those signals silently breaks. So:

- Introduce a normalized **`DeliveryAttemptFact`** in `scoring/events.py` — `{job_id, mode: tool_mediated|direct, model, passed, score, price_charged, duration}` — produced from **either** `delivered`+`delivery_*` (tool-mediated) **or** `submitted`+`verified_*` (direct). It carries the **realized** outcome only; hidden `pass_prob` is *not* in the trace. Every downstream signal (delivery pass-rate, gross score, support recovery, coherence "dropped vs delivered", realized net) consumes the fact, not the raw kind.
- **Expected-vs-realized separation (load-bearing for attribution).** The realized `passed` drives the **ledger / net revenue** (the brain lives with the actual draw). For **per-stage attribution** (tool-selection / pricing regret, §14), the scorer *joins* the fact with the menu's hidden `pass_prob` (reconstructed via provenance, §8B) and scores **expected value**, so a sound choice that fails the Bernoulli draw is not mislabeled a decision error. Do **not** derive tool-selection regret from realized `passed`.
- `deliver` must honor the **same guards `submit` does today** — most importantly "a pending manipulation must be resolved before delivery" (`env.py`) — and mark the job delivered so the coherence "dropped jobs" set stays correct.
- Net: the per-stage *formulas* are unchanged; they gain one normalization seam (realized for the ledger, EV for attribution) so both delivery paths feed them identically.

## 8. The Frozen Delivery Menu (PRD C.20.4–C.20.4.1)

`solvent/delivery/menu.py` + `menu_data/menu_v0_4.json`.

### 8.1 Schema (the interface-parity contract)

The same schema whether hand-authored (calibrated-synthetic) or `characterize`-produced:

```json
{
  "schema_version": "solvent_delivery_menu_v0_4",
  "source": "calibrated_synthetic | characterized",
  "tools": [
    {"name": "tool-mini", "price": 0.02, "speed_proxy": "fast",  "capability_proxy": 0.55},
    {"name": "tool-mid",  "price": 0.12, "speed_proxy": "med",   "capability_proxy": 0.80},
    {"name": "tool-pro",  "price": 0.45, "speed_proxy": "fast",  "capability_proxy": 0.95}
  ],
  "profile": {
    "data_clean": {
      "tool-mini": {"easy": {"pass": 0.97, "duration": 1}, "med": {"pass": 0.55, "duration": 2}, "hard": {"pass": 0.15, "duration": 4}},
      "tool-mid":  {"easy": {"pass": 0.99, "duration": 1}, "med": {"pass": 0.90, "duration": 2}, "hard": {"pass": 0.55, "duration": 3}},
      "tool-pro":  {"easy": {"pass": 1.00, "duration": 1}, "med": {"pass": 0.98, "duration": 1}, "hard": {"pass": 0.85, "duration": 2}}
    }
  }
}
```

`price` and `capability_proxy`/`speed_proxy` are **public** (returned by `list_models`); `profile` is **hidden** (the resolution table).

### 8.2 The four constraints (so it is not arbitrary — PRD C.20.4.1)

A loader-time validation enforces:

1. **Monotone difficulty** — within a tool, `pass` non-increasing and `duration` non-decreasing as difficulty rises.
2. **Non-dominated tool frontier** — no tool dominates on all of {cost, speed, reliability} for all difficulties (so tool choice is non-trivial).
3. **Magnitude calibration** — the shipped numbers are anchored to a small real measurement (run ~1–2 real models on ~5 real tasks once) and to public rates; recorded in `source`/provenance.
4. **Discriminating power** — a characterization check (C.12) confirms a good brain meaningfully beats a naive one and the loss localizes; floor/ceiling items pruned.

### 8.3 Lookup and sampling

`menu.resolve(task_type, model, internal_difficulty, draw_key) -> (passed, price, duration)` — a pure function of the menu + a **stable, call-order-independent draw key**, *not* a shared sequential RNG. Derive the key as `hash(episode_seed, job_id, model_name, attempt_index, menu_version)` (the same keyed-RNG pattern the market already uses, e.g. `random.Random(f"{seed}:{index}:…")`). This matters for two reasons:

- **Refactor-stability:** outcomes don't shift if the harness changes call order or the loop is restructured.
- **Paired-comparison fairness (load-bearing for attribution):** the *same job + same chosen tool* draws the *same* delivery luck regardless of which policy path reached it — so `naive` vs `procedure` differ because of their decisions, never because they happened to consume the RNG in a different order.

This is the only place delivery outcomes come from in tool-mediated mode (the realized slice of the menu — the `latent: GroundTruth` of C.4A.1).

## 8A. Task Corpus, Difficulty Parameterization & Menu Calibration

> The menu (§8) is a *mapping*; this section is **how that mapping is produced and how it maps to models** — the prerequisite §9 assumes. The §8.1 schema is only as credible as the task set used to anchor it. "Difficulty is a parameter, not a corpus" (C.20.4.1) is true for the *business loop*, but it **relocates** the corpus need into calibration — specified here.

### 8A.1 The mapping object

The menu **is** the model mapping. In v0.4a, delivery price is constant per tool (`tools[].price`) and difficulty only changes pass probability / duration:

```text
profile[task_type][difficulty][delivery_model] -> { pass_rate, mean_duration, n, ci }
tools[delivery_model].price -> price_charged
```

Two model-mappings exist and stay separate:

1. **Delivery tools → menu** — price/pass/duration per tool/task cell (the subject of this section).
2. **Brains → per-stage capability profile** — feeds the realizable reference (C.11.2); the policy under test, **not** in the menu.

### 8A.2 Three corpus tiers (different cost and rigor — do not conflate)

| Tier | What | Cost / rigor | v0.4 status |
|---|---|---|---|
| **0 — Generators** | `generate(seed, difficulty) -> (job, hidden truth, verifier)`; an infinite parameterized family the business loop draws from | cheap, ours | `data_clean` (add difficulty knob), `extract` (new) |
| **1 — Calibration corpus** | a *frozen sample* of generated instances, run against delivery models to populate the menu mapping | small real spend | **built in v0.4a** |
| **2 — Anchor corpus** | real tasks + verifiers for direct delivery / external validity | expensive | **SWE-Lancer, v0.5** — do not author our own |

### 8A.3 Difficulty parameterization (the real design work, per task type)

Each generator takes a difficulty knob that genuinely changes hardness and produces an **empirically monotone** pass-rate gradient across the tool tiers (validate, do not assume):

- `data_clean`: scale row count (e.g. 5 / 20 / 60), edge-case density (whitespace / case / invalid-plan / missing-cell transforms), spec ambiguity, distractor columns.
- `extract`: scale number of documents, distractor density, answer obfuscation, multi-hop.

Difficulty is scalar/tiered for the MVP (acknowledged simplification, C.20.4.2). The knob is what creates the "diverse range" — it must be designed and validated, not assumed.

### 8A.4 Calibration procedure (corpus → mapping)

1. **Freeze** a calibration corpus: `K ≈ 25` generated instances per `(task_type × difficulty)` cell, seeds recorded.
2. For each `delivery_model × cell × instance × R` reps: run delivery, run the **deterministic verifier** (§6 / §11), record `(passed, observed_cost, duration)`.
3. **Aggregate** → per-cell `(pass_rate, mean_duration)` with confidence intervals, and one calibrated public `price` per tool (for v0.4a, the menu schema keeps price tool-level rather than per-difficulty).
4. **Validate** the four menu constraints (§8.2: monotone, non-dominated frontier, calibration provenance, discriminating power) and **prune** floor/ceiling cells (C.12).
5. **Emit** the menu in the §8.1 schema (`source: characterized`).

### 8A.5 Sizing (why this is affordable)

2 types × 3 difficulties = 6 cells; 3–5 delivery models; `K ≈ 25`; `R ≈ 1–3` → ~**450–2,250 cheap solves**. For `data_clean` / `extract` that is seconds and pennies. `code_fix` is excluded *because* it is not cheap — it is the v0.5 anchor (SWE-Lancer), not something we generate.

### 8A.6 Delivery-model identity

Name menu entries as **specific real models, profiled once and frozen** (e.g. `tool-pro = <a frontier model>`) — realism *and* reproducibility against price/availability drift. Anonymized tiers (`tool-mini/mid/pro`) are an acceptable alternative; the §8.1 schema is identical either way.

### 8A.7 MVP shortcut (keeps v0.4a moving)

v0.4a may **ship a hand-authored menu calibrated against a minimal Tier-1 run** (1–2 models, ~5 instances/cell to anchor magnitudes) and defer full characterization. Interface parity (§8.1) makes the upgrade to a fully characterized menu a **data swap, not a code change**.

## 8B. Trace Provenance & Replay Contract

The v0.3 scorer reconstructs hidden ground truth by **rebuilding the `Market` from `episode_started` metadata** (seed + `market_version` + `market_size` + `decoy_rate`, in `scoring/scorecard.py`). Once delivery menus, difficulty, pricing, and a seed split exist, that metadata is **insufficient** — the scorer could silently reconstruct the *wrong* ground truth, and acceptance #10 ("re-score / replay makes no API calls") would be unmeetable. So `episode_started` must record a complete provenance block **before** any scoring change lands:

| Field | Why the scorer needs it |
|---|---|
| `seed`, `market_version`, `market_size`, `decoy_rate` | existing — rebuild the board |
| `menu_version` + `menu_checksum` | rebuild/verify the exact delivery menu used (outcomes depend on it) |
| `delivery_mode` (`tool_mediated`/`direct`) | choose the right `DeliveryAttemptFact` path (§7.5) |
| `task_mix` + `difficulty_distribution` | reconstruct per-job `internal_difficulty` ground truth |
| `seed_split` (`dev`/`test`) | self-describing results; never tune on test |
| `pricing_table_version` | reproduce the compute-economy figure exactly |
| `corpus_schema_version`, `menu_schema_version` | detect/upgrade format drift on replay |

Rules: the block is **deterministic** (no wall-clock, no absolute paths); the scorer **fails loudly** on a missing/mismatched checksum rather than guessing; re-scoring and `solvent replay` read this block + the recorded model sidecar (§6.4) and make **zero** API calls. Add this contract first, then build the scoring changes (§14) on top of it.

## 9. The Characterization Run (PRD C.12)

`solvent/characterize.py` + `solvent characterize`.

### 9.1 Two modes — `--validate-menu` (v0.4a) vs `--generate-menu` (v0.4-full)

`characterize` has two explicit subcommands so the v0.4a/v0.4-full split is unambiguous:

- **`solvent characterize --validate-menu` (the v0.4a gate).** Cheap, no menu generation: load the shipped hand-authored menu (§8A.7) and check it against the four constraints (§8.2) + floor/ceiling on the dev split. This is all v0.4a requires.
- **`solvent characterize --generate-menu --seeds dev` (the v0.4-full gate).** Runs the §8A.4 procedure end-to-end:
  1. Execute the calibration procedure over the Tier-1 corpus (§8A.2) → emit a menu in the **§8.1 schema** (`source: characterized`).
  2. Measure each **brain's** per-stage difficulty (pass rate per task type, achieved fraction-of-optimal, per-stage distributions) → the brains-to-capability mapping of §8A.1.
  3. **Prune** floor items (every config fails — suspect broken verifier) and ceiling items (every config passes — no information); flag saturation.

### 9.2 Wiring

- A generated menu drops into `delivery/menu_data/` and is used identically to the hand-authored one (interface parity).
- The per-brain profile feeds the **realizable reference** (PRD C.11.2): `realizable_reference` already accepts a capability term; extend it to consume the per-task profile.
- **v0.4a ships the hand-authored calibrated-synthetic menu and only runs `--validate-menu`** (§8A.7); `--generate-menu` is the v0.4-full upgrade path, not a v0.4a blocker.

### 9.3 Cost discipline (the "don't re-pay known costs" decision — PRD C.20.4)

Real delivery runs happen **only** during characterization. In comparison runs, delivery is **simulated from the frozen menu** (8.3), so a comparison pays only for the **brain's** reasoning tokens, not for re-solving tasks. This is the biggest cost lever and is most impactful for expensive task types.

## 12. Time Model — v0.4b (PRD C.20.5)

Built after v0.4a is stable.

- Time = a fixed horizon of simulated **business time**. **Advanced by work, not deliberation:** each `deliver` consumes `duration` from the chosen tool's profile (better tool → shorter); a background calendar drives job **arrival/expiry** and per-period overhead.
- **Deliberation is free in time** — the brain's chattiness must not move the business clock (explicitly *not* VB's tool-call-advances-time, which conflates verbosity with business time).
- Effect: faster tool → more throughput within the horizon → capacity utilization becomes a measurable skill; the brain trades **cost vs. speed vs. reliability** per job. With WIP = 1, this already yields the C.4A.2 attention-allocation tension (delivering one job lets another expire).

---

## 13. Reputation / Reactivity — v0.4b (PRD C.4A.3)

- A hidden `reputation` state variable, moved by delivery quality and manipulation outcomes, conditions the gigs sampled onto the future board (deliver well → more/higher-reservation gigs; concede freebies / drop jobs → fewer, worse).
- All draws come from the seeded `rng`, so a fixed policy reproduces exactly; reputation changes only *which* latent gigs surface and when. Keep `w_deliver / w_concede / w_drop` in the seed config (PRD C.4A.3).
- This is the piece that most makes the market feel alive and makes the long-horizon coherence axis bite — but it is intentionally last in v0.4 so it never destabilizes harness/economy bring-up.

## 14. Scoring & Comparison Changes

No existing per-stage formula changes. Additions only.

1. **Two economies** (PRD C.20.2): scorecard reports business `net_revenue` (revenue − overhead − delivery price) **and** a separate `ComputeEconomy` (brain tokens, brain cost) plus an **efficiency ratio** (e.g. `fraction_of_optimal_per_compute_dollar`). Compute cost is never in the death mechanic.
2. **Tool-selection signal — expected-value, not realized luck** (PRD C.20.3). `tool_selection_regret` is computed from **hidden expected value**, never from the realized Bernoulli draw, so the brain is judged on *decision quality* and not blamed for seeded bad luck. For each job, define `EV(tool) = reservation_price × pass_prob[tool][difficulty] − price[tool]` (− time cost in v0.4b); regret = `EV(EV-optimal tool) − EV(chosen tool)`. So **over-pay** = chose a tool whose marginal pass-probability gain didn't justify its extra price; **under-pay** = chose a tool whose *expected* clearance was too low for the job's value. A correct choice that happens to fail the draw has **zero** tool-selection regret — the failure shows up only in realized net revenue. The scorer computes `pass_prob` by joining the realized `DeliveryAttemptFact` (§7.5) with the menu it reconstructs from provenance (§8B); the **ledger/net uses realized pass/fail**, the **per-stage regret uses expected value**. (This mirrors how the omniscient optimum below is already EV-based — keeping the two consistent.)
3. **Optimum chooses the optimal tool per job** (PRD C.11.3): the omniscient optimum now, for each good job, selects the tool maximizing expected `(reservation_price × pass_prob − price)` under budget/horizon. Still computable because the env holds the menu. The **realizable** reference (C.11.2) conditions on capability; in tool-mediated mode delivery capability is the menu's (constant across brains), so the realizable gap concentrates in selection / pricing / tool-choice — exactly the firm/manager framing.
4. **Compare/viewer** gain the two-economy columns and the tool-selection row; the v0.3 labeling discipline (red-team off baseline vs. isolated manipulation loss) is preserved.

## 16. Testing Plan

### 16.1 Unit / integration (no live API)

- `test_llm_harness.py`: a `FakeClient` scripted to emit specific tool calls drives a full episode; assert the ReAct loop dispatches correctly, meters tokens, records the LLM sidecar, and that malformed calls become `invalid_action`.
- `test_delivery_menu.py`: schema load; the four constraints (monotone, non-dominated, calibration provenance present, lookup determinism); `resolve` is a pure seeded function.
- `test_two_economies.py`: business balance debits only overhead + delivery price; brain cost is reported but never debited; flat `tool_call_cost == 0` for LLM configs.
- `test_generator_difficulty_reproducible.py` (unit, no API): the difficulty knob (§8A.3) is reproducible by seed and produces the *structural* gradient it claims (e.g. more edge-cases / rows / distractors at higher tiers); the frozen Tier-1 corpus regenerates identically. (Does **not** assert a real pass-rate gradient — that needs delivery outcomes.)
- `test_tool_adapter.py` (unit): the §6.0 adapter validates tool calls against schema, serializes **public-only** observations, and turns each env exception (`InvalidActionError`/`AlreadyTerminatedError`/`UnknownJobError`) into a structured `invalid_action` observation rather than raising.
- `test_delivery_facts.py` (unit): `deliver` and `submit` paths both normalize into `DeliveryAttemptFact` (§7.5); delivery/gross/coherence/support compute identically from either; `deliver` rejects an unresolved pending manipulation.
- `test_trace_provenance.py` (unit): `episode_started` carries the §8B block; the scorer rebuilds ground truth from it and **fails loudly** on a menu-checksum mismatch.
- `test_characterize.py` (uses **recorded** delivery outcomes, no live API): `characterize` runs the §8A.4 procedure, emits a menu in the §8.1 schema + a per-stage brain profile, validates the four constraints, and flags floor/ceiling. The empirical pass-rate-gradient assertion lives **here**, on recorded data — not in the generator unit test.
- `test_extract.py`: generator + deterministic verifier; planted-answer match across difficulty tiers.
- `test_seed_split.py`: `--seeds dev|test` resolves correctly; split recorded in trace/scorecard.

### 16.2 Boundary / regression

- Extend the hidden-field guardrail to `internal_difficulty` and the per-model outcome table — assert absent from traces and viewer artifacts.
- Re-scoring a saved LLM trace makes **zero** API calls (replay from sidecar).
- **Stub trace compatibility (`test_stub_trace_compat.py`):** routing the stubs through the new §6.0 adapter must not change their behavior. Pin a golden set of v0.3 stub traces + scorecards and assert the post-adapter stubs reproduce them **semantically identical** (same event sequence, invalid-action payloads, burn timing, and scorecard) — byte-identical where feasible. This guards the refactor against silent event-order/payload/burn drift.
- All v0.1–v0.3 tests pass; the v0.3 stub demo + viewer still work; `stub:*` (and legacy `--agent stub --stub-mode`) run through the new dispatch unchanged.

### 16.3 Manual smoke (uses real API, dev seeds, small)

## 17. Implementation Sequence

**Build the §6.0 adapter first** (`env/tool_api.py`: tool schema, public observation serializer, action dispatcher, error envelope) and **refactor the stub to drive the env through it** — proving the boundary against the known-good stub before a model is involved. Then `model_client.py` (+ `FakeClient`/`RecordedClient`), the `LLMHarness` skeleton on the existing tool set (bid/submit), the §6.2 config-id grammar + `harness_from_config_id` dispatch + backward-compat, and token metering into a separate (not-yet-debited) compute figure. Acceptance: `test_tool_adapter.py`; a fake-client episode runs and scores like a stub; `stub:*` (and legacy `--agent stub --stub-mode`) unaffected.

### Step 2 — Trace provenance + delivery menu + tools + two economies + delivery facts
Land the §8B provenance block in `episode_started` and the `DeliveryAttemptFact` normalization (§7.5) **before** touching scoring. Then `delivery/menu.py` + `menu_v0_4.json` (stable keyed `resolve`, §8.3); `list_models` + `deliver` (honoring the pending-manipulation guard); debit delivery price, credit on pass; brain cost reported separately; flat tool cost → 0 for LLM. Update `optimal.py`/`scorecard.py` for per-job tool choice + tool-selection signal + compute economy, all consuming the normalized fact. Acceptance: `test_trace_provenance.py`, `test_delivery_facts.py`, `test_delivery_menu.py`, `test_two_economies.py`.

### Step 3 — Ablation knobs + prompts
`prompts.py`; `+memory / +planner / +procedure`. Acceptance: knobs measurably change policy on a fixed seed (fake + small real).

### Step 4 — Task corpus + difficulty parameterization (§8A)
Add the difficulty knob to `data_clean.py` and the `extract.py` generator + verifier (§8A.3); assert the knob yields a reproducible **structural** difficulty gradient (the empirical pass-rate gradient is validated later, in characterization); build the frozen Tier-1 calibration corpus (§8A.2). Acceptance: `test_generator_difficulty_reproducible.py`, `test_extract.py`.

### Step 5 — Characterization (corpus → menu) + dev/test split
`characterize.py` = the §8A.4 calibration procedure + brain profiling; `solvent characterize`; seed split files. Validate the four menu constraints and floor/ceiling pruning. v0.4a may ship the §8A.7 hand-authored menu and use `characterize` to validate it. Acceptance: `test_characterize.py`, `test_seed_split.py`.

### Step 6 — v0.4a regression + real smoke
Full suite green; hidden-field guardrail extended; replay-makes-no-API-calls; real-API smoke on dev seeds.

### Step 7 — v0.4b: real time, then reputation
Work-driven time + arrival/expiry calendar; then reputation conditioning the board. Each lands behind a config flag so v0.4a behavior is recoverable and any attribution shift is attributable.

## 18. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Real API cost/nondeterminism balloons the test + dev loop | Fake/recorded client in tests; record–replay so re-scoring is free; simulate delivery from the menu so only brain reasoning is paid; default to dev seeds + small N |
| Brain compute leaks into the death mechanic (re-confounding capability with provider price) | Hard separation: business balance debits only overhead + delivery price; compute economy is report-only (acceptance #4 + `test_two_economies.py`) |
| Delivery menu is arbitrary → degenerate or non-discriminating eval | The four loader-enforced constraints (8.2) + characterization discriminating-power check (C.12) + provenance in `source` |
| Hidden ground truth leaks (`internal_difficulty`, per-model table) | Public/hidden split enforced in tool returns; guardrail tests extended (16.2) |
| Tool-selection becomes trivial arithmetic | Exact cost but **noisy** capability proxy; hidden per-task outcome table; the agent must estimate (C.20.4.2) |
| Simulated delivery undercuts the "real verifier" guarantee | Determinism + gaming-resistance preserved by seeded draws; real verifiers retained in characterization + the v0.5 direct-delivery/anchor mode (7.3) |
| Scope creep: v0.4 tries to do harness + time + reputation at once | Strict staging (Section 4 / C.20.8); v0.4b behind flags; concurrency + anchor explicitly deferred to v0.5 |
| Stochastic policy makes comparisons noisy | Temp 0 + record/replay for reproducibility; `--samples N` + mean ± std, paired by seed (C.4A.4) |

## 19. Definition of Done

Two gates, matching the v0.4a / v0.4-full split (acceptance #7, §9.1).

**v0.4a Definition of Done** — the headline (real harness + delivery economy), shipping a hand-authored menu:

```bash
solvent characterize --validate-menu                 # validate the shipped menu; no menu generation
solvent compare --a claude-opus-4-8:base --b claude-opus-4-8:+procedure --seeds test --viewer
solvent view runs/<dir>
```

**v0.4-full Definition of Done** — v0.4a DoD plus corpus→menu generation and brain profiling:

```bash
solvent characterize --generate-menu --seeds dev     # full §8A.4 procedure → characterized menu
```

Across these gates, a reviewer should see:

1. A **real model** — not a stub — running the freelance business through the unchanged env boundary.
2. The agent **choosing which job to take, what to bid, and which delivery tool to hire**, against hidden ground truth it had to estimate.
3. A **business net** that charges delivery-tool compute but **not** the brain's own tokens, with brain compute shown separately as an efficiency figure.
4. A **per-stage scorecard** that now localizes loss across selection, pricing, **tool-selection / delivery**, support/manipulation, and coherence — for a real agent.
5. The `+procedure` ablation measurably changing the policy, paired by seed on the **test** split with mean ± std.
6. Re-scoring any trace with **zero** API calls.

That is v0.4: the attribution machine, now driven by a real swappable LLM harness under a realistic two-economy cost model — with reputation and real time (v0.4b) layered on once the harness and economy are proven, and concurrency, direct delivery, and the SWE-Lancer anchor deferred cleanly to v0.5.
